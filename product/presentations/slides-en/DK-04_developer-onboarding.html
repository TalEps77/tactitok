The Product

TactiTok in 2 Minutes

The Problem

  • No centralized, convenient way to deliver up-to-date training to fighters at the edge during operations
  • Face-to-face / ad-hoc delivery — PDFs in unsuitable channels — updates don't arrive in time
  • Limited bandwidth + kiosk environment makes video consumption cumbersome and inconsistent

The Solution

  • TikTok-style reels feed — short training videos, scroll to consume during rest breaks
  • Organized library — category tree + search for on-demand reference
  • Edge proxy cache — works offline; content cached locally via nginx in Docker
  • Admin portal — training staff upload content; edge syncs on reconnect

Flow A — Browse Reels

Open app → interest selection → scroll short-video feed → auto-play + prefetch → Like / Save / Download for offline

Flow B — Library Search

Enter library → navigate category tree or search → open content item (PDF or video) → view in browser → optional local download

Flow C — Admin Upload

Admin login → upload MP4 or PDF + metadata (title, categories, interests) → publish → edge syncs within 30 s → content visible on device

Edge Device Topology · DG-03

Why This Is Not a Normal Web App

Normal Web App

Browser → Cloud server (direct HTTPS)
Offline = Service Worker + Cache API
Complex offline code in the SPA
Browser caches content files (unreliable)

TactiTok Edge Device

Chrome → localhost:8080 (always)
Docker container intercepts all requests
nginx proxy_cache handles offline transparently
SPA has zero offline code — one code path

Offline Detection

SPA polls GET /api/health (never cached)
If it fails → cloud is unreachable
Response header X-Cache-Status: STALE = serving from cache
Show "Last synced" instead of "Synced now"

Edge Device Stack

[Windows PC]
  Chrome (kiosk mode) :localhost:8080
    ↕ localhost port mapping
[Linux VM (WSL2 or VirtualBox)]
  Docker container: tactitok-edge-proxy
    nginx :8080
    ├── /var/www/spa/ → Edge SPA bundle
    ├── /api/* → proxy_pass cloud
    └── proxy_cache: /var/cache/nginx (10 GB)

Data Model — Server · DG-04

Five PostgreSQL Entities, Content-Item Central

EntityKey Fields
ContentItem id (UUID PK), title, description, type (video|pdf), filename, fileSize, mimeType, duration (null for PDF), thumbnailPath, version, createdAt, updatedAt
Category id, name, parentId (self-ref FK, null = root), sortOrder — max 2 levels enforced in service layer
Interest id, name (UNIQUE), createdAt — flat list, no hierarchy
ContentCategory contentId (FK), categoryId (FK) — M:N junction; cascade delete on content delete
ContentInterest contentId (FK), interestId (FK) — M:N junction; cascade delete on content delete

Modeling Principles

  • One binary file per content item (no multi-file)
  • Hard delete — removes DB record + file + thumbnail
  • version counter, not version history
  • UUIDs for all PKs — future multi-source sync ready
  • Migration-based schema from Sprint 1 (Prisma or Knex)

Data Model — Edge Browser · DG-04

Four Local Entities in IndexedDB

DeviceProfile — singleton

deviceId: UUID (auto, stable across sessions)
selectedInterestIds: string[]
createdAt · updatedAt
Future: replace deviceId with userId when auth is added

CachedCatalog — singleton

items: ContentItemDTO[] — full server snapshot
categories: CategoryDTO[]
interests: InterestDTO[]
lastSyncedAt: ISO timestamp
Replaced entirely on each catalog pull

DownloadRecord — keyed by contentId

contentId · title · type · fileSize
downloadedAt · version (for "Updated" badge)
Metadata only — actual file is in nginx proxy_cache
Delete record ≠ delete file; proxy manages eviction

LocalAction — keyed by (contentId, action)

contentId · action: 'like' | 'save'
active: boolean · timestamp
No server sync in MVP
Designed for batch upload when user identity exists

Chrome Cache API / Service Workers: NOT used. All content caching is handled by nginx proxy_cache on the Linux VM filesystem. The SPA never stores binary file data.

Key API Principles

  • REST over JSON — consistent; no GraphQL overhead
  • ?v={version} on file URL = proxy cache-busting mechanism
  • Error envelope: {"error":"...","code":"..."}
  • ?since= accepted on catalog but ignored — ready for future delta sync
  • All types in packages/shared/src/types.ts
DG-02 — API Surface Diagram ↗

Architectural Decisions AD1–AD12

Twelve Decisions, All Deliberate

AD1 — TypeScript Monorepo

Shared types prevent API drift at compile time. Single CI pipeline.

AD2 — React for Both SPAs

Largest ecosystem; best scroll-snap + TikTok pattern support.

AD3 — Node.js + Express

Same language as frontend; simple range-request file streaming.

AD4 — PostgreSQL

Production-grade; ILIKE text search; migration-based from day one.

AD5 — Filesystem Storage Abstraction

store/retrieve/delete interface — swap to S3/MinIO without changing service layer.

AD6 — MP4 + HTTP 206 Range

No transcoding pipeline. Chrome handles range-based streaming natively. ≤3 min clips.

AD7 — nginx Proxy Cache (not Service Worker)

Zero offline code in SPA. Transparent offline for all previously accessed content.

AD8 — IndexedDB for State Only

Profile, download records, local actions. No binary file data. One SPA code path.

AD9 — Single Node.js Process

3 developers, ~20 users. Monolith with clean service modules. pm2 cluster if needed.

AD10 — nginx TLS on Cloud

Standard pattern. Let's Encrypt or self-signed for demo. proxy_pass :3000.

AD11 — Full Catalog Sync

~15 items = <10 KB JSON. API accepts ?since= for future delta sync.

AD12 — SPA Baked into Docker Image

SPA always available locally. Update = rebuild + redeploy Docker image to edge device.

Out of Scope

Explicit Deferrals + De-Scope Levers

Permanently Out of Scope (MVP)

  • User authentication / RBAC on edge (kiosk = anonymous)
  • Video transcoding pipeline (upload-ready MP4 only)
  • Advanced personalization / recommendation engine
  • Analytics dashboards / telemetry beyond local Like/Save
  • Chat, comments, social sharing features
  • Push sync / WebSocket (pull on open is sufficient)
  • CDN, S3, load balancer, Redis (single server, ~20 devices)
  • DRM, complex rights management
  • Service Workers / Cache API (nginx proxy replaces them)

Edge Proxy Deployment

Docker Build + Run, Auto-Restart on Boot

# On any machine with the repo: # 1. Build the Edge SPA bundle pnpm --filter client build # 2. Build Docker image (SPA bundle baked in) cd packages/edge-proxy docker build \ --build-arg SPA_DIST=../../packages/client/dist \ -t tactitok-edge-proxy:latest . # 3. Export image for transfer to edge device docker save tactitok-edge-proxy:latest \ | gzip > tactitok-edge-proxy.tar.gz # --- On the edge Linux VM --- # 4. Load the image docker load < tactitok-edge-proxy.tar.gz # 5. Run with auto-restart + persistent cache docker run -d \ --name tactitok-edge \ --restart=always \ -p 8080:8080 \ -v /var/tactitok-cache:/var/cache/nginx \ -e CLOUD_URL=https://your-vm.example.com \ tactitok-edge-proxy:latest # Verify curl http://localhost:8080/api/health

nginx.conf Key Settings

proxy_cache_path /var/cache/nginx
  max_size=10g inactive=30d;

# Catalog: 5-min TTL
/api/catalog → proxy_cache_valid 200 5m;

# Content files: 30-day TTL
/api/content/ → proxy_cache_valid 200 206 30d;

# Health: NEVER cache
/api/health → proxy_cache off;

# Serve SPA for all non-/api/ paths
/ → try_files $uri /index.html;

Update Procedure

1. Build new SPA bundle
2. Rebuild Docker image (bakes new bundle)
3. Transfer tar.gz to edge Linux VM
4. docker stop tactitok-edge
5. docker load < new.tar.gz
6. docker start tactitok-edge

What Comes After

Continuation Notes for the Next Team

Delta Sync (highest priority)

Replace full catalog pull with GET /api/catalog?since= returning only changed items + a deleted[] array. Server-side change only — SPA already sends the param.

User Authentication

DeviceProfile.deviceId becomes userId. API adds auth middleware to content endpoints. LocalAction records gain a synced flag for batch upload.

HLS Adaptive Streaming

For longer content or variable bandwidth: ffmpeg on upload generates HLS segments. Client swaps <video src> for hls.js manifest. Architecture unchanged otherwise.

Automated Test Coverage

MVP has no automated tests. Next team: unit tests for service layer (CatalogService, ContentFileService, AuthService) + integration tests for critical API endpoints before adding new features.

Monitoring + Observability

Add structured server-side request logging (pino) and error alerting (Sentry) before going to production with real users. Structured logs enable later analytics without schema changes.

Edge Proxy Proactive Prefetch

Add a cron job inside the Docker container that pulls all content from the cloud on a schedule. Ensures all content is offline-available before first user access — no waiting for the first request to cache.

Docker Compose

When edge device setup grows (multiple containers), replace docker run with a docker-compose.yml. Template should live in packages/edge-proxy/.
