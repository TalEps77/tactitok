3 server tables Direct assignment model Companion to DK-08

Overview

Schema at a glance

content_items

Core content metadata plus direct assignment fields:

  • category_id = single optional category
  • interest_ids = UUID array of interests

categories

2-level library hierarchy with self-reference via parent_id.

interests

Flat admin-managed lookup list used for personalization and reels filtering.

Delete semantics

  • Delete category → content becomes uncategorized
  • Delete interest → service strips UUID from arrays first

Key constraints

  • UNIQUE(parent_id, name) for sibling uniqueness
  • parent_id cascades to child categories
  • Depth limit of 2 is app logic, not a DB check

Important difference

Categories organize the library. Interests personalize discovery. The simplified schema keeps them conceptually separate even though both are now assigned directly from content_items.

Indexes and Rules

What matters operationally

Indexes

  • idx_content_type
  • idx_content_updated
  • idx_content_category
  • gin_content_interest_ids

Service rules

  • Validate, dedupe, and sort interest_ids
  • Remove deleted interest IDs from arrays before delete
  • Expose DTO field names as categoryId and interestIds

The important implementation tradeoff is simple: fewer tables in exchange for stricter service-layer discipline around interest arrays.
