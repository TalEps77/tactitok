Context

Why Security Matters Here

The Problem

  • Sensitive content — operational training material not for public access
  • No user identity — field kiosk is anonymous by design
  • Field device — not managed by the development team
  • No DRM — security relies on network isolation + TLS

MVP Security Scope

  • TLS on all cloud traffic — content never travels in plain text
  • Admin authentication — only authorised staff can publish content
  • No telemetry — no usage data leaves the edge device
  • Content validation — no executable files accepted
Out of scope for MVP: OS-level kiosk lockdown, VM firewall rules, physical device security, content DRM/classification — these are infrastructure responsibilities.

Trust Architecture

Three Trust Zones

Zone 1

Server — Trusted

Cloud VM: Content API, Admin API, PostgreSQL, file store. Authoritative source of truth. All data is here. HTTPS terminates here. Network access is restricted.

Zone 2

Edge Device — Semi-Trusted

Field kiosk: no user identity, anonymous access, read-only catalog. Chrome connects to edge proxy via localhost only. Local cache holds previously accessed content. Cannot mutate server-side data.

Zone 3

Admin Client — Authenticated

HQ desktop: shared password + session token, Admin API only, runs on HQ network. Session clears on tab close (sessionStorage, not localStorage).

Diagram reference: DG-16 (trust/security zones) provides a visual representation of these boundaries.

Zone 1 — Trusted

Zone 1 — Server

What runs here

  • Content API — read-only catalog + file serving for edge devices
  • Admin API — authenticated upload, metadata CRUD, category management
  • PostgreSQL — all catalog metadata (localhost only)
  • File store — uploaded videos and PDFs (localhost only)

Security properties

  • HTTPS termination — nginx on port 443; TLS 1.2+
  • DB + file store — localhost-only; never directly network-accessible
  • Authoritative — all content mutations happen here; edge is read-only
  • Network restriction — restrict to known IP range where possible (infra)

Content API: no auth needed

Edge devices read content anonymously. No auth middleware on /api/catalog or /api/content. Restriction is by network, not identity.

Zone 2 — Semi-Trusted

Zone 2 — Edge Device

Kiosk model — no identity

  • Anonymous access — no login, no user identity, device profile only
  • Read-only — can pull catalog + content, cannot mutate server data
  • Device profile — interest selection stored locally in IndexedDB; never sent to server

Network isolation

  • Chrome ↔ proxy — HTTP over localhost only (same machine, no network exposure)
  • Proxy ↔ cloud — HTTPS; proxy validates cloud TLS certificate
  • No direct cloud access from Chrome — proxy is the only egress point

What we control on edge

  • Docker container — nginx proxy + SPA bundle (team-owned)
  • Proxy cache — on Linux VM filesystem, not directly accessible from Windows
  • SPA code — all browser-side logic
Not controlled by team: Windows OS, Linux VM config, Docker daemon, kiosk mode setup, physical device access. Local browser storage is not encrypted — browser sandbox is the MVP boundary.

Zone 3 — Authenticated

Zone 3 — Admin Client

Authentication model

  • Shared password — single credential for all HQ admin staff (MVP)
  • Session token — returned on login; sent in Authorization header on every Admin API call
  • sessionStorage — token clears automatically on tab close
  • Token expiry — server-side; expired tokens are rejected

Access scope

  • Admin API only — upload, metadata CRUD, category management
  • HQ network — admin portal not intended to be public-facing
  • No RBAC — single role; per-user permissions are post-MVP
Continuation note: The shared password + single role model is deliberately simple for MVP. A post-MVP upgrade path would add per-user accounts and RBAC using the same Admin API — no structural changes needed.

Transport Security

Transport Security

Connection Protocol Notes
Chrome → Edge Proxy HTTP localhost only — same physical machine; no network exposure
Edge Proxy → Cloud Server HTTPS TLS 1.2+ Proxy validates cloud TLS cert; rejects invalid certs
HQ Desktop → Cloud Server HTTPS TLS 1.2+ Standard browser TLS; admin portal served over HTTPS
Cloud: nginx port 443 TLS Termination nginx terminates TLS; forwards to Node.js on localhost:3000

TLS Certificate (demo)

  • Preferred: Let's Encrypt (free, auto-renew, if domain available)
  • Fallback: Self-signed cert (acceptable for lab/demo environment)

Why HTTP on localhost is acceptable

  • Traffic stays within the same physical machine
  • No network interface is crossed — loopback only
  • Consistent with standard browser/proxy architecture

Content Security

Content Security

Upload validation (server)

  • MIME type + file extension — both validated on upload; mismatch rejected
  • No executables — only MP4 video and standard PDF accepted
  • Size limit — files >100 MB rejected
  • Generated filenames — stored as {contentId}.{ext}; no user-supplied paths

File serving

  • No direct filesystem URLs — files served only via API endpoint
  • API controls headers — Content-Type, Accept-Ranges, Cache-Control set by server
  • No path traversal risk — content-id is validated before file lookup

Edge cache isolation

  • Proxy cache on Linux VM filesystem — not accessible from Windows side
  • Chrome accesses cached files only via HTTP through proxy — never directly
  • IndexedDB — stores metadata records only; no raw file data in browser storage

No telemetry

  • Zero usage data leaves the edge device in MVP
  • Like/Save actions stored locally in IndexedDB only — no server sync
  • No analytics platform integrated

Security Scope

MVP Scope — What We Do NOT Cover

Infrastructure responsibilities

  • OS-level kiosk config — Chrome kiosk mode and Windows lockdown are pre-configured by the deployment team; not a team deliverable
  • VM firewall rules — cloud VM network security groups / iptables rules are the infrastructure team's responsibility
  • Physical device security — securing the tablet against physical access is an operational responsibility

Post-MVP scope

  • Content DRM / classification — no digital rights management; content protection relies on network isolation + TLS only
  • Encryption of local browser storage — IndexedDB and proxy cache are not encrypted; browser sandbox is the MVP boundary
  • Per-user authentication on edge — field kiosk is deliberately anonymous in MVP
  • Audit logging — no access audit trail beyond standard server request logs
For reviewers: This is a lab prototype for a demo environment. The security model is appropriate for that context. A production deployment would require hardening in the areas listed above.

Demo Environment

Demo Environment Posture

Lab / demo setup

  • Cloud server — reachable on known network; HTTPS with self-signed or Let's Encrypt cert
  • Edge devices (×N) — each running Docker container with nginx proxy + SPA bundle; Chrome connects to localhost:8080
  • Admin portal — accessible to reviewers on HQ network; not publicly reachable
  • Demo content — 10 PDFs + 5 videos pre-loaded; catalog synced to edge on first launch

What reviewers can verify hands-on

  • HTTPS lock icon in browser on cloud server
  • Chrome address bar shows localhost — no direct cloud URL
  • Admin portal login required before any content mutation
  • Uploading a non-MP4/non-PDF file is rejected
  • Disconnecting from network — previously accessed content still plays
  • Browser DevTools: no external network requests from edge device (no telemetry)
