DK-08 · DB Implementation PRD

Implementing the 3-table server schema

This deck translates the revised DB PRD into build-ready implementation guidance: 3 PostgreSQL tables, direct category_id assignment, array-based interest_ids, and clear service-layer cleanup rules.

Source: 05_data-model.md Source: 08_db-implementation-prd.md Version 1.1 April 23, 2026

Architecture

What changed in the implementation model

Old normalized shape

  • 5 server tables
  • Two assignment tables for category and interest links
  • DB-level integrity for both dimensions

New MVP shape

  • content_items, categories, interests
  • content_items.category_id is a nullable FK
  • content_items.interest_ids is a PostgreSQL UUID[]

Integrity split

  • Category validity stays in the database
  • Interest validity moves to application services
  • Delete-interest cleanup becomes explicit service behavior

Why this is acceptable for MVP

  • Fewer tables and fewer ORM relationships
  • Simpler migration story for a 3-person team
  • Still continuation-friendly if normalization is needed later

Models

ContentItem category_id, interest_ids, file metadata, version, timestamps Use MutableList.as_mutable(ARRAY(UUID)) so in-place list edits are tracked.
Category parent_id, sort_order, timestamps Self-referential FK with cascade on child categories; depth limit remains app logic.
Interest name, created_at No ORM link table. Service code owns cleanup of array references.
category_id = mapped_column( UUID(as_uuid=True), ForeignKey("categories.id", ondelete="SET NULL"), nullable=True, ) interest_ids = mapped_column( MutableList.as_mutable(ARRAY(UUID(as_uuid=True))), nullable=False, server_default="{}", )

Services

Where the application layer now matters most

Catalog writes

  • Validate every UUID in interestIds
  • Reject unknown interests before save
  • Dedupe and sort array values for stable writes

Category delete

  • Child categories still cascade through categories.parent_id
  • Affected content becomes uncategorized automatically
  • No content rows are deleted by category removal

Interest delete

  • Run array cleanup before deleting the interest row
  • Use PostgreSQL array_remove in a bulk update
  • Edge-selected IDs are cleaned on next sync

Tests that must exist

  • Model defaults and index presence
  • Interest array validation and normalization
  • Category delete sets category_id to null
  • Interest delete removes UUIDs from all arrays

Definition of Done

Implementation checklist

Must be true

  • 3-table schema matches the PRD and API contract
  • No DTO still exposes a plural category array field
  • No code assumes separate assignment tables still exist
  • Delete semantics are tested end to end

Still out of scope

  • User accounts
  • Soft delete
  • Richer metadata on interest assignment
  • Normalization beyond what MVP needs

If future analytics or richer assignment metadata demand it, normalize later. The MVP implementation now optimizes for delivery speed and clarity.
