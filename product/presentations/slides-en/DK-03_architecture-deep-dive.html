Architecture Goals

Six Goals That Drive Every Decision

AG1 — Simplest Viable Architecture

3 student developers, 8–10 weeks. Minimize moving parts. Single language. No microservices.

AG2 — End-to-End Functional

Upload → catalog → stream → download → offline must all work. No half-built flows in the MVP.

AG3 — Continuation-Ready

Clean layer separation. A follow-on team can extend any layer without rewriting the others.

AG4 — Single Tech Stack

TypeScript everywhere — frontend and backend. Shared types. Reduced cognitive overhead.

AG5 — Offline-Aware from Day One

Edge proxy + local SPA bundle designed into the architecture. Not bolted on later.

AG6 — Demo-Optimized

Fast video start, smooth scrolling, reliable offline. These user-facing requirements drive architectural choices.

System Boundaries

What's In Scope — What's Out

✓ In Scope — Built by the Team

  • Edge SPA — Reels, Library, Downloads (HTML/TS)
  • Edge Proxy — Docker/nginx; SPA serving, proxy cache, offline
  • API Server — Node.js/Express; Content API, Admin API
  • Admin SPA — Upload, catalog, categories (HTML/TS)
  • PostgreSQL — Metadata, categories, interests
  • File Store — MP4 + PDF on server filesystem

✗ Out of Scope — Explicitly Deferred

  • User identity / authentication — edge is anonymous (kiosk)
  • Video transcoding — upload-ready MP4 only
  • Analytics / telemetry — local Like/Save only
  • Push sync — pull-based on app open
  • CDN, S3, load balancer — single server, ~20 devices
  • DRM, RBAC, monitoring — network isolation is the boundary

Architecture Decision AD7

Why nginx Caching Instead of Service Worker

✗ Service Worker approach — Rejected

  • ✗ Complex browser-side caching code in SPA
  • ✗ Cache API management — quota, eviction, versioning
  • ✗ Two code paths: online vs offline
  • ✗ Cache invalidation is the SPA's responsibility
  • ✗ Doesn't cache video range-request responses well
  • ✗ Hard to debug; HTTPS required even in dev

✓ nginx proxy_cache — Chosen

  • ✓ Zero caching code in SPA
  • ✓ Identical code path online and offline
  • ✓ nginx handles eviction, TTL, stale serving
  • ✓ X-Cache-Status header surfaces HIT / MISS / STALE
  • ✓ Caches full video files — range requests served from cache
  • ✓ 10 GB local cache on Linux VM filesystem

"The SPA's online code and offline code are identical. The proxy handles everything."

SPA bundle serving API + content proxying Offline cache serving

Edge Proxy Configuration

Cache Settings That Drive Offline Behaviour

proxy_cache_path /var/cache/nginx
  max_size=10g  inactive=30d;

# Content files — rarely change
location /api/content/ {
  proxy_pass https://CLOUD_SERVER;
  proxy_cache_valid 200 30d;
  proxy_cache_use_stale
    error timeout updating
    http_502 http_503 http_504;
  add_header X-Cache-Status
    $upstream_cache_status;
}

# Catalog metadata — shorter TTL
location /api/catalog {
  proxy_pass https://CLOUD_SERVER;
  proxy_cache_valid 200 5m;
  proxy_cache_use_stale
    error timeout updating;
}

# SPA — served from Docker image
location / {
  root /var/www/spa;
  try_files $uri $uri/ /index.html;
}
SettingValueWhy
Max cache size10 GB~15 items × ≤100 MB fits
Content file TTL30 daysRarely changes; ?v= cache-busts
Catalog TTL5 minutesFrequent enough; stale OK offline
Stale servingerror, timeout, 5xxOffline mode on any cloud failure

Cache Invalidation

Content URLs include ?v={version}. When admin updates a video, the version increments. New cache key = cache MISS = fresh file fetched from cloud. No manual purge needed.

Content & Data Flow

Three Key Flows

Flow A — Admin Upload

1Admin uploads MP4 / PDF via Admin SPA
2API Server validates: MIME, size ≤100 MB
3File stored: ./data/content/{uuid}.ext
4Metadata inserted into PostgreSQL (version=1)

Flow B — Catalog Sync

1Edge SPA opens → GET /api/catalog → :8080
2Proxy: MISS → forward to cloud → cache 5 min
3SPA stores catalog snapshot in IndexedDB
4Client-side filtering for Reels and Library

Flow C — Video Prefetch (Reels)

1User watches video N (from proxy cache)
2At 30%: SPA sends range request for video N+1
3Proxy fetches full file from cloud → caches 30 d
4Swipe → instant playback from local cache

DG-04 · Data Model — Server

Five Entities in PostgreSQL

5 server entities · UUIDs for all PKs · Hard delete in MVP · Migration-based schema (Prisma / Knex)

Data Model — Edge Browser

Four Local Entities in IndexedDB

DeviceProfile — singleton

deviceId (auto UUID, stable across sessions)
selectedInterestIds: string[]
createdAt · updatedAt
Continuation: replace deviceId with userId when auth added

CachedCatalog — singleton

items: ContentItemDTO[] — full server catalog
categories: CategoryDTO[] · interests: InterestDTO[]
lastSyncedAt: ISO timestamp
Replaced entirely on each sync pull

DownloadRecord — keyed by contentId

contentId · title · type · fileSize
downloadedAt · version (for "updated" badge)
Metadata only — actual file is in nginx proxy_cache
Delete record ≠ delete file (proxy manages eviction)

LocalAction — keyed by (contentId, action)

contentId · action: 'like' | 'save'
active: boolean · timestamp
No server sync in MVP
Designed for future batch upload when identity exists

v0.2 change: DownloadRecord no longer stores a cache key or blob reference. Chrome Cache API is not used. All content caching is handled by nginx proxy_cache on the Linux VM filesystem.

Architectural Decisions

Part 1 — Foundation Choices

AD1 — TypeScript Monorepo

Single language. Shared types between client and server prevent API contract drift at compile time. One CI pipeline.

AD2 — HTML + TypeScript for Both SPAs

No framework dependency. Simpler stack. Sufficient for SPA complexity. Avoids React learning curve.

AD3 — Node.js + Express

Same language as frontend. Simple file streaming (206 range). Adequate for ~20 concurrent clients.

AD4 — PostgreSQL

Production-grade. Text search (ILIKE for MVP; tsvector later). Concurrent reads. Migration-based schema from day one.

AD5 — Filesystem for Content

Simplest. No external service. store/retrieve/delete abstraction allows S3/MinIO swap without changing service layer.

AD6 — MP4 + HTTP Range (206)

No transcoding pipeline. Chrome handles range-based streaming natively. ≤3 min clips are sufficient for MVP.

Architectural Decisions

Part 2 — Offline & Deployment Choices

AD7 — nginx Proxy Cache

Replaces Service Worker + Cache API. Zero offline code in SPA. Transparent offline for all previously accessed content.

AD8 — IndexedDB for State Only

Profile, download records, local actions. All content caching delegated to proxy. SPA code stays simple — one code path.

AD9 — Single Node.js Process

3 developers, 10 weeks, ~20 users. Monolith with clean internal service modules. pm2 cluster available if needed.

AD10 — nginx TLS (Cloud)

Standard pattern. Offloads TLS from Node. Let's Encrypt or self-signed for demo. proxy_pass to localhost:3000.

AD11 — Full Catalog Sync

~15 items = <10 KB JSON. Delta sync adds complexity for no benefit. API accepts ?since= for future delta.

AD12 — SPA in Docker Image

SPA always available locally. No cloud dependency for app startup. Update = rebuild + redeploy Docker image.

Continuation Notes

How a Follow-on Team Extends This

  • User identity: Content API pre-wires auth middleware but bypasses it for edge routes. Add passport.js; swap deviceId for userId in DeviceProfile. No other changes needed.
  • Object storage (S3/MinIO): ContentFileService exposes store / retrieve / delete. Replace the filesystem implementation — service layer and API do not change.
  • CDN: Content proxy already serves with Cache-Control and ETag headers. CDN integration is a configuration change, not a code change.
  • Push sync (WebSocket/SSE): Express server supports WS alongside REST. Design a SyncProvider interface so pull can be swapped for push without SPA rewrites.
  • Analytics: LocalActions store deviceId + contentId + timestamp. Batch-sync to server when identity exists. IndexedDB schema is ready — no changes needed.
  • Proactive edge prefetch: Add a background job to the proxy that pulls all new content into cache on catalog change. Currently caches on first access; proactive prefetch covers all content before first user request.
