TactiTok Execution Plan

How 3 developers deliver the MVP in 12 weeks · milestones · risks · fallbacks

3 Developers 6 Sprints · 12 Weeks 12 Acceptance Criteria

March 2026 · Source: product/07_delivery-plan.md

Scope

What We're Delivering

Flow A — Browse Reels

TikTok-style vertical feed · auto-play · swipe navigation · interest-based filtering · Like / Save / Download for offline

Flow B — Library Search

Category tree navigation · full-text search · in-browser video player and PDF viewer · optional local download

Flow C — Admin Upload

Secure admin login · upload MP4 or PDF with metadata · category & interest tagging · edge syncs within 30 s

Definition of Done — 12 Acceptance Criteria

  • End-to-end demo flow completes (all 15 steps)
  • Reels auto-play and swipe work smoothly on tablet
  • Video playback starts ≤ 2 s on reasonable network
  • Library search returns results within 5 s
  • PDF renders correctly in-browser for 3+ test documents
  • Downloaded video plays offline with zero network
  • Downloaded PDF renders offline with zero network
  • Admin upload → edge visibility ≤ 30 s after sync
  • Interest selection filters reels feed correctly
  • 15 demo items (10 PDFs + 5 videos) loaded and navigable
  • All interaction inside Chrome — no pop-ups, no external apps
  • App handles network drop gracefully — no crashes

Sprint 1 · Weeks 1–2

Foundation — Everyone Unblocked

Dev A — Backend Foundation

Monorepo setup (pnpm workspaces) · shared TypeScript types · PostgreSQL schema (migration 001) · Express server skeleton · admin auth (JWT) · upload & list endpoints

Dev B — Edge SPA Skeleton

HTML + TypeScript + Vite in packages/client · 3-tab bar (Reels / Library / Downloads) · routing (History API) · global CSS baseline · IndexedDB setup · DeviceProfile init

Dev C — Admin SPA + Edge Proxy

HTML + TypeScript + Vite in packages/admin · login screen wired to JWT endpoint · Dockerfile + nginx.conf for edge proxy · proxy starts and serves on localhost:8080

Sprint 1 Exit Criteria

  • pnpm install && pnpm build succeeds across all packages
  • Cloud VM has PostgreSQL running; migration 001 applied
  • GET /api/health returns 200
  • Admin login returns a JWT token
  • Edge proxy Docker container starts and proxies GET /api/health from cloud
  • Admin SPA login screen reaches the server
  • Edge SPA loads on Chrome via localhost:8080

Milestone M1: End of week 2 — monorepo builds; DB schema exists; all 3 apps start; edge proxy proxies successfully

Sprint 2 · Weeks 3–4

Content Pipeline — Upload to Edge

Dev A

GET /api/catalog (full catalog response) · GET /api/content/:id/file with range request support (206) · thumbnail endpoint · full CRUD for categories and interests · file replacement + version increment

Dev B

Edge SPA catalog sync → IndexedDB · library flat list with content cards · in-browser video player (HTML5) · PDF viewer via PDF.js · manual refresh button

Dev C

Admin upload form (file picker, title, description, category/interest selectors, progress bar) · admin content list (sortable) · edit form · delete with confirmation · category and interest CRUD UI · seed data script

Sprint 2 Exit Criteria

  • Upload a video via admin → appears in GET /api/catalog
  • Upload a PDF via admin → appears in GET /api/catalog
  • Edge SPA syncs catalog → items appear in library flat list
  • Video file streams with range requests (206 response confirmed)
  • PDF opens and renders in-browser from edge SPA
  • Delete content in admin → removed from catalog on next sync
  • Edge proxy caches content file (X-Cache-Status: HIT on 2nd request)

Milestone M2: End of week 4 — upload → catalog → stream/view on edge device; categories and interests exist

Sprint 3 · Weeks 5–6

Reels Experience — Highest-Risk Sprint

⚠ Why this is the highest-risk sprint

Requires scroll-snap + IntersectionObserver + touch gestures + Chrome auto-play policy + video prefetch to all work together on the actual tablet hardware. Must be validated hands-on by end of week 6 (M3). If it fails, de-scope levers are pulled immediately.

Dev B (primary) + Dev C (support)

Full-screen vertical feed (CSS scroll-snap) · auto-play via IntersectionObserver · swipe gestures (touchstart/touchend) · tap-to-start overlay for first video · overlay UI (title, Like, Save, Download) · video prefetch at 30% playback · interest-based filtering · interest selection screen · feed empty state

Dev C

Library category tree (2-level collapsible) · content type badges · text search bar (client-side filtering) · download button on library cards · empty state messages

Sprint 3 Exit Criteria

  • Reels feed scrolls smoothly on 10″ tablet (no jank; confirmed hands-on)
  • First video starts within ≤ 2 s on reasonable network
  • Auto-play works after first user gesture
  • Swipe up → next video; swipe down → previous video
  • Interest selection: select 2 interests → feed shows only matching videos
  • Library category tree: tap category → filtered list
  • Text search: type term → filtered list within 1 s
  • Video prefetch: swipe to next video starts faster (qualitative)

Milestone M3: End of week 6 — TikTok-style feed validated hands-on on 10″ tablet. If M3 is at risk → pull de-scope levers immediately.

Sprint 4 · Weeks 7–8

Library, Downloads & Offline

Dev B

Download flow (fetch → proxy cache → IndexedDB DownloadRecord) · download progress bar · Downloads tab (list with title, type, size, date) · play/view downloaded content via proxy cache · delete from Downloads · network status indicator (poll /api/health) · offline messaging in Reels and Library · orphan cleanup on sync

Dev C

"Updated" badge: compare DownloadRecord version with catalog version · Download button state (Downloaded vs. Download) · PDF page navigation (previous/next, counter) · standalone video player view · content detail view consistency across Reels / Library / Downloads

Dev A

Nginx offline test: stop cloud server → confirm SPA loads, catalog serves stale cache, files serve from cache · document nginx config for stale serving · load test: 5 concurrent video streams; switch to nginx X-Accel-Redirect if needed · verify Accept-Ranges, Content-Length, ETag headers

Sprint 4 Exit Criteria

  • Download a video → "Downloaded" icon → appears in Downloads tab
  • Download a PDF → appears in Downloads tab
  • Disconnect network → Downloads tab fully functional
  • Offline: play downloaded video → plays without network
  • Offline: open downloaded PDF → renders without network
  • Offline indicator appears in Reels and Library when disconnected
  • "Updated" badge appears after admin updates a downloaded item
  • Delete downloaded item → removed from Downloads tab
  • Orphan cleanup: delete content in admin → sync → item removed

Milestone M4: End of week 8 — download video + PDF → disconnect → play/view offline; "Updated" badge works

Dev C — Admin Portal Final Polish

Category CRUD: reorder, 2-level limit UI · interest CRUD: final polish · content list: sort and filter by type · upload form: client-side validation, clear error messages · content update workflow

Dev B — Demo Walkthrough

Full 15-step demo script walkthrough on actual device · fix UX issues found · measure video playback start time · tune prefetch if needed

Sprint 5 Exit Criteria

  • All 21 API endpoints respond correctly (manual integration test)
  • Full demo script runs end-to-end without errors on staging
  • AC-1 through AC-12 documented; all "Must" criteria green or escalated
  • Cloud VM deployed and reachable via HTTPS
  • Edge proxy Docker image builds and runs on edge device

Milestone M5: End of week 10 — all 15 demo steps run end-to-end; all AC pass or are escalated to stakeholder

Post-MVP

Continuation Roadmap — After MVP

1. Delta Sync Priority 1

Replace full catalog pull with delta-based response using ?since={timestamp}. Server returns only changed items + deleted array. Highest-priority scalability improvement as catalog grows.

2. User Authentication

Edge SPA has no user identity in MVP. When auth is added, DeviceProfile.deviceId becomes userId; API adds auth middleware; LocalAction gains a synced flag for batch upload.

3. HLS Adaptive Streaming

MP4 + range requests works for ≤ 3 min clips. For longer content or variable bandwidth, add HLS segmentation (ffmpeg on upload) and an HLS player (hls.js). The <video> element can be swapped without changing the rest of the architecture.

4. Edge Proactive Prefetch

Add a scheduled job inside the Docker container (cron) that pulls all content into cache on a schedule — rather than waiting for the first user request. Ensures all content is available offline even before first access.

5. Test Coverage

MVP has no automated tests. Next team should add unit tests for the service layer (CatalogService, ContentFileService, AuthService) and integration tests for critical API endpoints before adding new features.

6. Monitoring

Add structured server-side request logging (pino) and error alerting (Sentry) before going to production with real users. Set up the Grafana or equivalent dashboard for latency and error rate.
