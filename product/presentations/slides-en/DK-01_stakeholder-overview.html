DK-01 — Stakeholder Overview

TactiTok:
Training at the Edge

Browser-based training delivery for field personnel
under constrained connectivity

Evaluator Briefing · 2026

Operational Problem

The Training Gap at the Edge

🚫 No Centralized Delivery

There is no convenient, reliable way to deliver up-to-date training content to personnel at the edge during operations. The content exists — it just doesn't reach the field.

❌ Face-to-Face Training Fails

Today's model relies on in-person sessions and ad-hoc file sharing in unsuitable channels. Updates arrive late — or not at all. Trainers must be physically present.

📶 Bandwidth-Constrained Consumption

Under limited bandwidth and a kiosk-like setup, consuming video and documents is cumbersome and inconsistent. Buffering, failures, and poor UX break the learning experience.

🔄 No Habitual Pull Mechanism

There is no experience that mirrors natural short-form content consumption — nothing that makes learning feel routine during rest breaks. Learning requires deliberate effort every time.

The Vision

"Turn wartime downtime into short, accessible, up-to-date operational learning — even under non-ideal connectivity."

TikTok for Training

  • Fighters scroll a curated feed of short video clips (≤3 min), filtered by their selected interests
  • Videos auto-play as they appear — no deliberate search required
  • Learning becomes a natural habit during rest breaks, not a task to schedule

Library & Hybrid Offline

  • For specific questions, a searchable category library provides targeted lookup
  • Metadata syncs when network is available; content is fetched on demand
  • Manual download guarantees offline access for any item — video or PDF

Operational Context

Designed for the Edge

💻

Hardware

  • ~10″ Windows tablet
  • Chrome in kiosk mode
  • Ruggedized computing environment
  • Sometimes projected for a small team
⚙️

Runtime Stack

  • Windows host → Linux VM (WSL2)
  • Docker container running nginx
  • Edge proxy serves app at localhost:8080
  • Chrome connects only to localhost
📡

Connectivity

  • Unstable / limited bandwidth
  • Hybrid: metadata syncs when online
  • Content fetched on demand via proxy
  • Manual download for guaranteed offline

ⓘ  Chrome never connects directly to the cloud — all traffic routes through the local edge proxy, which caches everything it fetches.

System Overview

How It Works

Chrome never connects directly to the cloud — the edge proxy handles all routing and caching.

Offline Architecture

The Offline Challenge — Solved

📌 Smart Proxy Cache

nginx caches every file it fetches. Up to 10 GB stored on the device, 30-day TTL. Downloads trigger a full cache load automatically.

⚡ Transparent Offline

The SPA code is identical online and offline. No Service Worker, no offline-specific code. Proxy serves stale cache; SPA reads an HTTP header to show the offline indicator.

⇩ Manual Download

One tap in reels or library. Record stored in IndexedDB; file pre-loaded in proxy cache. Accessible in Downloads tab — video and PDF fully offline.

Success Metrics

Seven Criteria for Demo Success

ID Criterion Target
S1 End-to-end flow Upload → publish → catalog → consume (video + PDF) → download → view offline
S2 Library lookup time ≤ 60 seconds to find a specific item in a demo scenario
S3 Reels playback start ≤ 2 seconds on reasonable network; fast-start via prefetch on weak network
S4 Browser containment 100% of viewing inside Chrome — no external apps, no pop-ups
S5 Interest filtering ≥ 3 selectable interests that consistently filter feed and library
S6 Offline playback Previously downloaded video and PDF viewable with zero network
S7 Updated badge Downloaded items show “updated” badge when catalog version exceeds download version after sync

If all 7 criteria pass → MVP accepted for continuation development.

Structural success (S8–S10: code quality, developer onboarding) validated separately by supervisors.

Post-MVP Roadmap

Designed to Continue

Each MVP architectural decision was made to enable clean extension without major rewrites:

  • User identity & RBAC — The MVP is userless (kiosk). A device-id pattern allows adding authentication without schema changes.
  • Recommendation engine — Interest filtering is the MVP. View counts and like counts are tracked locally, ready to feed a future algorithm.
  • Server-side analytics — Like/Save are local-only in MVP. The local action log is designed for batch-sync when analytics are added.
  • Multi-language support — English only in MVP. All UI strings use a key-based system; adding a language is a configuration change.
  • Push-based sync — MVP uses pull on app open. The API is designed so a WebSocket channel can be added alongside the existing REST endpoints.

"Clean layered architecture — a follow-on team can extend any layer without touching the others."

● Content Management Admin portal, upload, categories
● Catalog / API Node.js, PostgreSQL, REST
● Delivery / Offline nginx proxy, cache, sync
● UI React SPA, reels, library
