Overview

9 Entities — Server vs. Edge

■ SERVER — PostgreSQL (Cloud VM)

Authoritative source of truth · 5 tables

ContentItem Category Interest ContentCategory (junction) ContentInterest (junction)

■ EDGE — IndexedDB (Chrome Browser)

Device-local state · 4 object stores

DeviceProfile (singleton) CachedCatalog (singleton) DownloadRecord (keyed by contentId) LocalAction (composite key)
No User table. Admin auth uses a shared JWT password (no user record). Edge identity is a device-scoped UUID in DeviceProfile, not server-persisted.

Server Side

Server-Side Entities

PostgreSQL · 5 Tables

  • content_items — central entity; 1 row per content piece
  • categories — 2-level hierarchy (parentId nullable)
  • interests — flat tags; admin-managed
  • content_categories — M:N junction
  • content_interests — M:N junction

File Storage (Filesystem, not DB)

./data/ ├── content/ │ ├── {uuid}.mp4 │ └── {uuid}.pdf └── thumbnails/ └── {uuid}.jpg

Server Entity 2/3

Category — 2-Level Hierarchy

FieldTypeConstraints
idUUIDPK, auto-generated
nameVARCHAR(100)NOT NULL, UNIQUE within siblings
parentIdUUIDNULLABLE — FK → Category.id
sortOrderINTEGERNOT NULL, default 0
createdAtTIMESTAMPNOT NULL, auto
updatedAtTIMESTAMPNOT NULL, auto

Hierarchy Rule

parentId IS NULL → root category (level 1) parentId IS NOT NULL → child (level 2) Max depth = 2 (app logic, not DB) Child's parentId must reference a row where parentId IS NULL

Example Tree

Combat Techniques (parentId = NULL) └─ Unarmed (parentId = "uuid-combat") └─ Knife Defense (parentId = "uuid-combat") Field Medicine (parentId = NULL)
Delete cascade: Deleting a parent cascades to child categories and removes all junction rows (ContentCategory). Content items themselves are not deleted — they become uncategorized.

Server Entity 3/3

Interest — Flat Tags

FieldTypeConstraintsNotes
idUUIDPK, auto-generatedReferenced in DeviceProfile.selectedInterestIds
nameVARCHAR(100)NOT NULL, UNIQUE (global)Display name; shown in interest selection UI
createdAtTIMESTAMPNOT NULL, auto
No hierarchy. Interests are a flat list — no parent/child relationship. If the admin creates "Combat", there are no sub-interests. Use categories for hierarchy; use interests for cross-cutting filters.

Role in the System

  • Admin creates interests via Admin SPA
  • Admin tags content items with interests (ContentInterest junction)
  • Edge device user selects interests on first launch
  • Selected IDs stored in DeviceProfile.selectedInterestIds
  • Reels feed filters by matching interestIds from CachedCatalog

On Delete (Interest)

All ContentInterest junction rows cascade-deleted. Edge device profiles that reference the deleted interest ID become stale — cleaned up on next catalog sync by the edge client.

Server Junctions

Junction Tables — M:N Relationships

ContentCategory

ContentItem ↔ Category (many-to-many)

FieldTypeConstraints
contentIdUUIDFK → ContentItem.id, CASCADE
categoryIdUUIDFK → Category.id, CASCADE

PK: composite (contentId, categoryId)

Zero categories allowed: A content item can belong to zero categories (uncategorized). It will appear under "All" in the library.

ContentInterest

ContentItem ↔ Interest (many-to-many)

FieldTypeConstraints
contentIdUUIDFK → ContentItem.id, CASCADE
interestIdUUIDFK → Interest.id, CASCADE

PK: composite (contentId, interestId)

Cascade Behavior

  • Delete ContentItem → all junction rows removed
  • Delete Category → ContentCategory rows removed
  • Delete Interest → ContentInterest rows removed
  • No orphaned junction rows possible

Server ERD

Server ER Diagram

Category (self 1:N via parentId) │1 │N Category (child) ContentItem ──── M:N ──── Category │ (ContentCategory) │ └──────── M:N ──── Interest (ContentInterest) Cardinalities: ContentItem → categories : 0..N Category → content items : 0..N ContentItem → interests : 0..N Interest → content items : 0..N Category → children : 0..N (max 1 level)
Edge references: DeviceProfile → Interest, DownloadRecord → ContentItem, and LocalAction → ContentItem are UUID-only references — no DB foreign keys. Cleaned up by sync logic.

Edge Side

Edge-Side Entities

IndexedDB · 4 Object Stores

Store Key Records
deviceProfile'default'1 (singleton)
catalogCache'current'1 (singleton)
downloadscontentId0 – 15+
localActions[contentId, action]0 – 30+

What is NOT in IndexedDB

  • Content binary files (MP4, PDF) — cached by edge proxy nginx
  • Thumbnail images — served directly from proxy cache
  • SPA static bundle — built into Docker image, served by nginx
IndexedDB total size: < 50 KB for 15 content items. Tiny. All the heavy storage is in the nginx proxy_cache on the Linux VM filesystem (1–1500 MB for video/PDF files).

Sizing (15 demo items)

IndexedDB (all stores): < 50 KB · Edge proxy cache (files): 15 – 1500 MB · SPA bundle (Docker): ~5–10 MB

Edge Entity 1/4

DeviceProfile — Singleton

FieldTypeNotes
deviceIdstring (UUID)Auto-generated on first app open. Stable across sessions.
selectedInterestIdsstring[]Array of Interest UUIDs from CachedCatalog. Drives reels feed.
createdAtISO timestampFirst-time setup date
updatedAtISO timestampLast interest change
Singleton: IndexedDB key is the string 'default'. Only one record ever exists. No list, no pagination.

Lifecycle

  • First app open: check if 'default' exists in IndexedDB
  • If not found → show interest selection screen
  • User selects interests → write DeviceProfile with new UUID + selectedInterestIds
  • On interest update → update selectedInterestIds + updatedAt only
  • deviceId is never regenerated

Continuation Note

deviceId can later be replaced by or linked to a userId when authentication is added. The field name and structure remain the same; only the generation logic changes.

Edge Entity 3/4

DownloadRecord — Metadata Only

FieldTypeNotes
contentIdstring (UUID)IndexedDB key — FK-like ref to ContentItem.id
titlestringSnapshot of title at download time
type'video' | 'pdf'Snapshot of type at download time
fileSizenumberBytes
downloadedAtISO timestampWhen file was downloaded
versionnumberVersion at download time — compare with catalog for "updated" badge

Where is the File?

  • File is in nginx proxy_cache on the Linux VM filesystem
  • SPA requests file via GET /api/content/{id}/file
  • Edge proxy serves from cache (HIT) or fetches from cloud (MISS)
  • DownloadRecord tells the UI "user explicitly downloaded this"

"Updated" Badge Logic

Compare DownloadRecord.version with CachedCatalog.items[].version (matched by contentId). If catalog version is higher → show "UPDATED" badge on the Downloads tab.

Delete behavior: Removing DownloadRecord removes only IndexedDB metadata. Proxy cache is unaffected — nginx manages eviction (30-day inactivity rule).

Edge Entity 4/4

LocalAction — Like & Save

FieldTypeNotes
contentIdstring (UUID)Part of composite key
action'like' | 'save'Part of composite key
activebooleantrue = currently liked/saved; false = toggled off
timestampISO timestampLast toggle time
Composite key: IndexedDB key is [contentId, action]. One record per (content, action) pair. Toggling updates active and timestamp — no duplicates accumulate.

Toggle Logic

// Pseudo-code: like button tap const key = [contentId, 'like']; const existing = await db.localActions.get(key); if (existing) { existing.active = !existing.active; existing.timestamp = new Date().toISOString(); await db.localActions.put(existing); } else { await db.localActions.add({ contentId, action: 'like', active: true, timestamp: new Date().toISOString() }); }

Continuation Note

When user identity is added: batch-sync all records where active = true using deviceId + contentId + timestamp. Add a synced flag to avoid re-syncing.

Server ↔ Edge

What Lives Where & Why

Data Location Why Sync behavior
Content metadata (title, type, version…)Server (PostgreSQL)Authoritative; admin-managed; must survive device lossPulled to CachedCatalog on sync
Content binary files (MP4, PDF)Server FS + proxy cacheToo large for IndexedDB; served via edge proxyProxy caches on first request; 30-day eviction
Category & interest definitionsServer (PostgreSQL)Admin-managed; reference dataIncluded in catalog pull
Device interest selectionEdge (IndexedDB)Personal preference; no user identity; no server recordNot synced — local only
Catalog snapshotEdge (IndexedDB)Enables offline browsing and filtering without networkFull replace on GET /api/catalog
Download metadata (which items downloaded)Edge (IndexedDB)UI state only; file is in proxy cache, not tracked server-sideNot synced — local only
Like / Save actionsEdge (IndexedDB)No user identity; local-only engagement in MVPNot synced in MVP

Sync Flow: Catalog Pull

SPA → GET /api/catalog → proxy (cache MISS or TTL expired) → cloud API → PostgreSQL JOIN → JSON response → proxy caches 5 min → SPA writes CachedCatalog singleton

Sync Flow: Download Trigger

User taps Download → SPA fetches full file via proxy → proxy caches → SPA writes DownloadRecord with version snapshot → Downloads tab updates

Summary

Summary & Next Steps

Key Takeaways

  • 9 entities total: 5 server (PostgreSQL) + 4 edge (IndexedDB)
  • Server is authoritative. Edge is a cached mirror + local-only state.
  • No User table in MVP. Device identity is local-only UUID.
  • Files never in IndexedDB. nginx proxy_cache handles all file storage.
  • Version counter (not hash) drives the "updated" badge on downloads.
  • Hard delete everywhere. No soft-delete, no draft state in MVP.
  • UUIDs as PKs. Enables future multi-source sync without collision.
  • Shared TS types in packages/shared bridge server ↔ edge.

Next Steps

  • Implement Prisma schema matching these 5 tables + constraints

  • Define shared TypeScript DTO interfaces in packages/shared
