הבעיה

  • אין דרך מרכזית ונוחה להעביר תכנים עדכניים לאנשי הקצה בשטח במהלך מבצעים

הפתרון

  • פיד ריאלס בסגנון TikTok — סרטוני אימון קצרים, גלילה לצריכה בהפסקות

  • מטמון פרוקסי קצה — עובד ללא רשת; תוכן שמור מקומית דרך nginx ב-Docker
  • פורטל מנהל — צוות הדרכה מעלה תוכן; הקצה מסתנכרן עם חזרת הרשת

טופולוגיית מכשיר הקצה · DG-03

למה זו אינה אפליקציית ווב רגילה

אפליקציית ווב רגילה

דפדפן → שרת ענן (HTTPS ישיר)
אופליין = Service Worker + Cache API
קוד אופליין מורכב ב-SPA
הדפדפן מאחסן קבצי תוכן (לא אמין)

מכשיר קצה TactiTok

Chrome → localhost:8080 (תמיד)
מכיל Docker מיירט את כל הבקשות
proxy_cache של nginx מטפל באופליין שקוף
ב-SPA אין קוד אופליין — נתיב קוד אחד

זיהוי מצב אופליין

ה-SPA מבצע פולינג ל-GET /api/health (אף פעם לא במטמון)
אם נכשל → הענן לא נגיש
כותרת X-Cache-Status: STALE = הגשה מהמטמון
הצג "סונכרן לאחרונה" במקום "מסונכרן עכשיו"

מחסנית מכשיר הקצה

[Windows PC]
  Chrome (kiosk mode) :localhost:8080
    ↕ localhost port mapping
[Linux VM (WSL2 or VirtualBox)]
  Docker container: tactitok-edge-proxy
    nginx :8080
    ├── /var/www/spa/ → Edge SPA bundle
    ├── /api/* → proxy_pass cloud
    └── proxy_cache: /var/cache/nginx (10 GB)

DG-03 — פריסת מכשיר קצה ↗

מודל נתונים — שרת · DG-04

חמש ישויות PostgreSQL, תוכן-פריט במרכז

ישותשדות מרכזיים
ContentItem id (UUID PK), title, description, type (video|pdf), filename, fileSize, mimeType, duration (null עבור PDF), thumbnailPath, version, createdAt, updatedAt
Category id, name, parentId (FK עצמי-ייחוסי, null = שורש), sortOrder — מקסימום 2 רמות נאכף בשכבת השירות
Interest id, name (UNIQUE), createdAt — רשימה שטוחה, ללא היררכיה
ContentCategory contentId (FK), categoryId (FK) — צומת M:N; מחיקה בהתחשלות עם מחיקת תוכן
ContentInterest contentId (FK), interestId (FK) — צומת M:N; מחיקה בהתחשלות עם מחיקת תוכן

עקרונות מידול

  • קובץ בינארי אחד לכל פריט תוכן (ללא ריבוי קבצים)
  • מחיקה קשה — מסירה רשומת DB + קובץ + תמונה ממוזערת
  • מונה גרסה, לא היסטוריית גרסאות
  • UUIDs לכל ה-PK — מוכן לסנכרון ממקורות מרובים בעתיד
  • סכמה מבוססת-מיגרציה מספרינט 1 (Prisma או Knex)
DG-04 — מודל נתונים מלא ↗

מודל נתונים — דפדפן קצה · DG-04

ארבע ישויות מקומיות ב-IndexedDB

DeviceProfile — יחיד

deviceId: UUID (אוטומטי, יציב בין סשנים)
selectedInterestIds: string[]
createdAt · updatedAt
עתיד: החלף deviceId ב-userId עם הוספת אימות

CachedCatalog — יחיד

items: ContentItemDTO[] — תמונת שרת מלאה
categories: CategoryDTO[]
interests: InterestDTO[]
lastSyncedAt: חותמת זמן ISO
מוחלף כולו בכל משיכת קטלוג

DownloadRecord — ממוין לפי contentId

contentId · title · type · fileSize
downloadedAt · version (לתג "עודכן")
מטא-דאטה בלבד — הקובץ עצמו ב-nginx proxy_cache
מחיקת רשומה ≠ מחיקת קובץ; הפרוקסי מנהל פינוי

LocalAction — ממוין לפי (contentId, action)

contentId · action: 'like' | 'save'
active: boolean · timestamp
ללא סנכרון שרת ב-MVP
מתוכנן להעלאה אצווה כאשר זהות משתמש קיימת

Chrome Cache API / Service Workers: לא בשימוש. כל מטמון התוכן מנוהל על ידי nginx proxy_cache במערכת הקבצים של Linux VM. ה-SPA לעולם אינו מאחסן נתוני קובץ בינארי.

עקרונות API מרכזיים

  • REST על JSON — עקבי; ללא עומס GraphQL
  • ?v={version} בכתובת קובץ = מנגנון cache-busting של פרוקסי
  • מעטפת שגיאה: {"error":"...","code":"..."}
  • ?since= מתקבל בקטלוג אך מתעלם — מוכן לסנכרון דלתא עתידי
  • כל הטיפוסים ב-packages/shared/src/types.ts
DG-02 — דיאגרמת משטח API ↗

החלטות ארכיטקטורה AD1–AD12

שתים-עשרה החלטות, כולן מכוונות

AD1 — מונורפו TypeScript

טיפוסים משותפים מונעים סחף API בזמן קומפילציה. צינור CI אחד.

AD2 — HTML + TypeScript לשני ה-SPAs

ללא תלות ב-framework; מחסנית פשוטה יותר; מספיקה למורכבות SPA.

AD3 — Node.js + Express

אותה שפה כמו הפרונטאנד; שידור קבצי range-request פשוט.

AD4 — PostgreSQL

דרגת ייצור; חיפוש טקסט ILIKE; מיגרציה-מבוססת מהיום הראשון.

AD5 — הפשטת אחסון מערכת קבצים

ממשק store/retrieve/delete — מחליף ל-S3/MinIO מבלי לשנות שכבת השירות.

AD6 — MP4 + HTTP 206 Range

ללא צינור קידוד. Chrome מטפל בשידור range-based באופן מובנה. קליפים ≤3 דקות.

AD7 — מטמון פרוקסי nginx (לא Service Worker)

אפס קוד אופליין ב-SPA. אופליין שקוף לכל תוכן שנגיש בעבר.

AD8 — IndexedDB למצב בלבד

פרופיל, רשומות הורדה, פעולות מקומיות. ללא נתוני קובץ בינארי. נתיב קוד SPA אחד.

AD9 — תהליך Node.js יחיד

3 מפתחים, ~20 משתמשים. מונוליט עם מודולי שירות נקיים. pm2 cluster במידת הצורך.

AD10 — nginx TLS בענן

תבנית סטנדרטית. Let's Encrypt או חתום-עצמי לדמו. proxy_pass :3000.

AD11 — סנכרון קטלוג מלא

~15 פריטים = <10 KB JSON. ה-API מקבל ?since= לסנכרון דלתא עתידי.

AD12 — SPA מוטמע בתמונת Docker

ה-SPA תמיד זמין מקומית. עדכון = בנייה מחדש + פריסת Docker לקצה.

מחוץ לתחום

דחיות מפורשות + מנופי צמצום

מחוץ לתחום לצמיתות (MVP)

  • אימות משתמש / RBAC בקצה (קיוסק = אנונימי)
  • צינור קידוד וידאו (MP4 מוכן להעלאה בלבד)
  • התאמה אישית מתקדמת / מנוע המלצות
  • לוחות מחוונים אנליטיים / טלמטריה מעבר ל-Like/Save מקומי

  • CDN, S3, load balancer, Redis (שרת יחיד, ~20 מכשירים)
  • DRM, ניהול זכויות מורכב
  • Service Workers / Cache API (nginx proxy מחליף אותם)

פריסת פרוקסי קצה

Docker Build + Run, הפעלה אוטומטית בעת אתחול

# על כל מכונה עם הריפו: # 1. בנה חבילת Edge SPA pnpm --filter client build # 2. בנה תמונת Docker (חבילת SPA אפויה פנימה) cd packages/edge-proxy docker build \ --build-arg SPA_DIST=../../packages/client/dist \ -t tactitok-edge-proxy:latest . # 3. ייצא תמונה להעברה למכשיר קצה docker save tactitok-edge-proxy:latest \ | gzip > tactitok-edge-proxy.tar.gz # --- על Linux VM בקצה --- # 4. טען את התמונה docker load < tactitok-edge-proxy.tar.gz # 5. הרץ עם הפעלה מחדש אוטומטית + מטמון קבוע docker run -d \ --name tactitok-edge \ --restart=always \ -p 8080:8080 \ -v /var/tactitok-cache:/var/cache/nginx \ -e CLOUD_URL=https://your-vm.example.com \ tactitok-edge-proxy:latest # אמת curl http://localhost:8080/api/health

הגדרות מרכזיות ב-nginx.conf

proxy_cache_path /var/cache/nginx
  max_size=10g inactive=30d;

# Catalog: 5-min TTL
/api/catalog → proxy_cache_valid 200 5m;

# Content files: 30-day TTL
/api/content/ → proxy_cache_valid 200 206 30d;

# Health: NEVER cache
/api/health → proxy_cache off;

# Serve SPA for all non-/api/ paths
/ → try_files $uri /index.html;

נוהל עדכון

1. בנה חבילת SPA חדשה
2. בנה מחדש תמונת Docker (אופה חבילה חדשה)
3. העבר tar.gz ל-Linux VM בקצה
4. docker stop tactitok-edge
5. docker load < new.tar.gz
6. docker start tactitok-edge

מה מגיע אחר-כך

הערות המשך לצוות הבא

סנכרון דלתא (עדיפות גבוהה ביותר)

החלף משיכת קטלוג מלאה ב-GET /api/catalog?since= המחזיר רק פריטים שהשתנו + מערך deleted[]. שינוי בצד השרת בלבד — ה-SPA כבר שולח את הפרמטר.

אימות משתמש

DeviceProfile.deviceId הופך ל-userId. ה-API מוסיף middleware אימות לנקודות קצה של תוכן. רשומות LocalAction מקבלות דגל synced להעלאת אצווה.

HLS Adaptive Streaming

לתוכן ארוך יותר או רוחב פס משתנה: ffmpeg בהעלאה מייצר פלחי HLS. הלקוח מחליף <video src> ב-hls.js manifest. הארכיטקטורה אינה משתנה אחרת.

כיסוי בדיקות אוטומטי

ל-MVP אין בדיקות אוטומטיות. הצוות הבא: בדיקות יחידה לשכבת השירות (CatalogService, ContentFileService, AuthService) + בדיקות אינטגרציה לנקודות קצה קריטיות לפני הוספת תכונות חדשות.

ניטור + יכולת תצפית

הוסף רישום בקשות מובנה בצד השרת (pino) והתראת שגיאות (Sentry) לפני מעבר לייצור עם משתמשים אמיתיים. לוגים מובנים מאפשרים אנליטיקה מאוחרת ללא שינויי סכמה.

שליפה מוקדמת יזומה בפרוקסי קצה

הוסף cron job בתוך מכיל ה-Docker שמושך את כל התוכן מהענן לפי לוח זמנים. מבטיח שכל התוכן זמין ללא רשת לפני גישת המשתמש הראשון — ללא המתנה לבקשה הראשונה לאחסן.

Docker Compose

כאשר הגדרת מכשיר הקצה גדלה (מכילים מרובים), החלף docker run ב-docker-compose.yml. התבנית צריכה לשכון ב-packages/edge-proxy/.
