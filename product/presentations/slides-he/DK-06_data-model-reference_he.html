Category — היררכיה דו-רמתית

שדהסוגאילוצים
idUUIDPK, נוצר אוטומטית
nameVARCHAR(100)NOT NULL, ייחודי בתוך אחים
parentIdUUIDניתן לריק — FK → Category.id
sortOrderINTEGERNOT NULL, ברירת מחדל 0
createdAtTIMESTAMPNOT NULL, אוטומטי
updatedAtTIMESTAMPNOT NULL, אוטומטי

כלל היררכיה

parentId IS NULL → קטגוריית שורש (רמה 1) parentId IS NOT NULL → ילד (רמה 2) עומק מקסימלי = 2 (לוגיקת אפליקציה, לא DB) parentId של ילד חייב להפנות לשורה שבה parentId IS NULL

דוגמת עץ

טכניקות לחימה (parentId = NULL) └─ חסימה ידנית (parentId = "uuid-combat") └─ הגנת סכין (parentId = "uuid-combat") רפואת שדה (parentId = NULL)
מחיקה בcascade: מחיקת הורה מתגלגלת לקטגוריות ילד ומסירה את כל שורות הצומת (ContentCategory). פריטי תוכן עצמם אינם נמחקים — הם הופכים לא מסווגים.

Interest — תגיות שטוחות

שדהסוגאילוציםהערות

ללא היררכיה. תחומי עניין הם רשימה שטוחה — אין קשר הורה/ילד. השתמשו בקטגוריות להיררכיה; בתחומי עניין למסנני חיתוך.

תפקיד במערכת

  • מנהל יוצר תחומי עניין דרך ה-Admin SPA
  • מנהל מתייג פריטי תוכן בתחומי עניין (צומת ContentInterest)
  • משתמש מכשיר קצה בוחר תחומי עניין בהפעלה ראשונה
  • מזהים שנבחרו נשמרים ב-DeviceProfile.selectedInterestIds
  • פיד ריאלס מסנן לפי interestIds תואמים מ-CachedCatalog

במחיקה (Interest)

כל שורות צומת ContentInterest נמחקות בcascade. פרופילי מכשיר קצה שמפנים למזהה העניין שנמחק הופכים מיושנים — יפונו בסנכרון קטלוג הבא על-ידי לקוח הקצה.

התנהגות Cascade

  • מחיקת ContentItem → כל שורות הצומת מוסרות
  • מחיקת Category → שורות ContentCategory מוסרות
  • מחיקת Interest → שורות ContentInterest מוסרות
  • אין שורות צומת עם הפניה לא חוקית אפשריות

ERD שרת

דיאגרמת ER — שרת

Category (self 1:N via parentId) |1 |N Category (child) ContentItem ---- M:N ---- Category | (ContentCategory) | +-------- M:N ---- Interest (ContentInterest) קרדינליות: ContentItem → categories : 0..N Category → content items : 0..N ContentItem → interests : 0..N Interest → content items : 0..N Category → children : 0..N (מקס רמה 1)
הפניות קצה: DeviceProfile → Interest, DownloadRecord → ContentItem, ו-LocalAction → ContentItem הן הפניות UUID בלבד — ללא FK של מסד נתונים. מנוקות על-ידי לוגיקת סנכרון.

מה לא נמצא ב-IndexedDB

  • קבצים בינאריים של תוכן (MP4, PDF) — מאוחסנים במטמון על-ידי פרוקסי nginx
  • תמונות ממוזערות — מוגשות ישירות ממטמון הפרוקסי
  • חבילת ה-SPA הסטטית — מובנית בתוך תמונת Docker, מוגשת על-ידי nginx
גודל IndexedDB כולל: < 50 KB עבור 15 פריטי תוכן. זעיר. כל האחסון הכבד נמצא ב-nginx proxy_cache במערכת הקבצים של ה-VM של Linux (1–1500 MB לקבצי וידאו/PDF).

גודל (15 פריטי הדגמה)

IndexedDB (כל המאגרים): < 50 KB · מטמון פרוקסי קצה (קבצים): 15 – 1500 MB · חבילת SPA (Docker): ~5–10 MB

DeviceProfile — יחיד (Singleton)

שדהסוגהערות
deviceIdstring (UUID)נוצר אוטומטית בפתיחה ראשונה. יציב בין סשנים.
selectedInterestIdsstring[]מערך של UUID-ים של Interest מ-CachedCatalog. מניע פיד ריאלס.

יחיד: מפתח IndexedDB הוא המחרוזת 'default'. קיימת רשומה אחת בלבד לעולם. אין רשימה, אין עימוד.

מחזור חיים

  • פתיחה ראשונה: בדוק אם 'default' קיים ב-IndexedDB
  • אם לא נמצא → הצג מסך בחירת תחומי עניין
  • משתמש בוחר תחומי עניין → כתוב DeviceProfile עם UUID חדש + selectedInterestIds
  • בעדכון תחומי עניין → עדכן selectedInterestIds + updatedAt בלבד
  • deviceId לעולם לא נוצר מחדש

הערת המשכיות

deviceId יכול מאוחר יותר להיות מוחלף על-ידי או מקושר ל-userId כשמוסיפים אימות. שם השדה והמבנה נשארים זהים; רק לוגיקת היצירה משתנה.

DownloadRecord — מטאדטה בלבד

שדהסוגהערות
contentIdstring (UUID)מפתח IndexedDB — הפניה דמוית FK ל-ContentItem.id
titlestringצילום כותרת בזמן ההורדה

versionnumberגרסה בזמן ההורדה — השווה עם קטלוג לתג "עודכן"

היכן הקובץ?

  • הקובץ נמצא ב-proxy_cache של nginx במערכת הקבצים של ה-VM
  • ה-SPA מבקש קובץ דרך GET /api/content/{id}/file
  • פרוקסי הקצה מגיש ממטמון (HIT) או שולף מהענן (MISS)
  • DownloadRecord אומר לממשק "משתמש הוריד זאת במפורש"

לוגיקת תג "עודכן"

השווה DownloadRecord.version עם CachedCatalog.items[].version (לפי contentId תואם). אם גרסת הקטלוג גבוהה יותר → הצג תג "עודכן" בלשונית הורדות.

התנהגות מחיקה: הסרת DownloadRecord מסירה מטאדטת IndexedDB בלבד. מטמון הפרוקסי לא מושפע — nginx מנהל פינוי (כלל 30 יום חוסר פעילות).

LocalAction — לייק ושמירה

שדהסוגהערות
contentIdstring (UUID)חלק מהמפתח המורכב
action'like' | 'save'חלק מהמפתח המורכב

מפתח מורכב: מפתח IndexedDB הוא [contentId, action]. רשומה אחת לכל זוג (תוכן, פעולה). הפעלה מעדכנת active ו-timestamp — לא מצטברות כפילויות.

לוגיקת הפעלה

// Pseudo-code: like button tap const key = [contentId, 'like']; const existing = await db.localActions.get(key); if (existing) { existing.active = !existing.active; existing.timestamp = new Date().toISOString(); await db.localActions.put(existing); } else { await db.localActions.add({ contentId, action: 'like', active: true, timestamp: new Date().toISOString() }); }

הערת המשכיות

כשמוסיפים זהות משתמש: סנכרן אצווה את כל הרשומות שבהן active = true באמצעות deviceId + contentId + timestamp. הוסף דגל synced למניעת סנכרון חוזר.

סיכום

סיכום וצעדים הבאים

נקודות מפתח

  • 9 ישויות בסך הכל: 5 שרת (PostgreSQL) + 4 קצה (IndexedDB)
  • השרת הוא סמכותי. הקצה הוא מראה מאוחסנת + מצב מקומי בלבד.
  • ללא טבלת User ב-MVP. זהות מכשיר היא UUID מקומי בלבד.
  • קבצים לעולם לא ב-IndexedDB. nginx proxy_cache מטפל באחסון קבצים.
  • מונה גרסה (לא גיבוב) מניע תג "עודכן" על הורדות.
  • מחיקה קשיחה בכל מקום. ללא מחיקה רכה, ללא מצב טיוטה ב-MVP.
  • UUID כ-PK. מאפשר סנכרון ממקורות מרובים עתידי ללא התנגשות.
  • טיפוסי TS משותפים ב-packages/shared גושרים שרת ↔ קצה.

צעדים הבאים

  • מימוש סכמת Prisma התואמת 5 טבלאות + אילוצים אלו
  • כתיבת מיגרציות לכל ישויות השרת לפני כל קוד API
  • הגדרת ממשקי TypeScript DTO משותפים ב-packages/shared
