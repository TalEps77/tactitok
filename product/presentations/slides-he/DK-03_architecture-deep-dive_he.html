החלטת ארכיטקטורה AD7

מדוע nginx Caching ולא Service Worker

✗ גישת Service Worker — נדחתה

  • ✗ קוד caching מורכב בצד ה-SPA
  • ✗ ניהול Cache API — מכסה, פינוי, גרסאות
  • ✗ שני נתיבי קוד: מחובר לעומת אופליין
  • ✗ ביטול cache הוא אחריות ה-SPA
  • ✗ לא מטמין תגובות range-request לוידאו היטב
  • ✗ קשה לדיבוג; נדרש HTTPS גם בפיתוח

✓ nginx proxy_cache — נבחר

  • ✓ אפס קוד caching ב-SPA
  • ✓ נתיב קוד זהה מחובר ואופליין
  • ✓ nginx מטפל בפינוי, TTL, הגשה מעותקת
  • ✓ כותרת X-Cache-Status מציגה HIT / MISS / STALE
  • ✓ מטמין קבצי וידאו מלאים — range requests מוגשים מ-cache
  • ✓ 10 GB cache מקומי במערכת הקבצים של Linux VM

"קוד האונליין וקוד האופליין של ה-SPA הם זהים. ה-proxy מטפל בהכל."

הגשת חבילת SPA Proxy ל-API ותוכן הגשת cache אופליין

תצורת Edge Proxy

הגדרות Cache שמניעות התנהגות האופליין

proxy_cache_path /var/cache/nginx
  max_size=10g  inactive=30d;

# Content files — rarely change
location /api/content/ {
  proxy_pass https://CLOUD_SERVER;
  proxy_cache_valid 200 30d;
  proxy_cache_use_stale
    error timeout updating
    http_502 http_503 http_504;
  add_header X-Cache-Status
    $upstream_cache_status;
}

# Catalog metadata — shorter TTL
location /api/catalog {
  proxy_pass https://CLOUD_SERVER;
  proxy_cache_valid 200 5m;
  proxy_cache_use_stale
    error timeout updating;
}

# SPA — served from Docker image
location / {
  root /var/www/spa;
  try_files $uri $uri/ /index.html;
}
הגדרהערךסיבה
גודל cache מקסימלי10 GB~15 פריטים × ≤100 MB מתאים
TTL קובצי תוכן30 יוםמשתנה לעתים רחוקות; ?v= מבטל cache
TTL קטלוג5 דקותמספיק תכוף; stale בסדר באופליין

ביטול Cache

URLs של תוכן כוללות ?v={version}. כאשר admin מעדכן וידאו, הגרסה עולה. מפתח cache חדש = cache MISS = קובץ חדש מהענן. אין צורך בביטול ידני.

ארבע ישויות מקומיות ב-IndexedDB

מוחלף לגמרי בכל משיכת סנכרון

DownloadRecord — ממוין לפי contentId

contentId · title · type · fileSize
downloadedAt · version (לתג "עודכן")
מטא-דאטה בלבד — הקובץ בפועל נמצא ב-nginx proxy_cache
מחיקת רשומה ≠ מחיקת קובץ (proxy מנהל פינוי)

LocalAction — ממוין לפי (contentId, action)

contentId · action: 'like' | 'save'
active: boolean · timestamp
אין סנכרון שרת ב-MVP
מעוצב להעלאה אצווית עתידית כשתהיה זהות

שינוי v0.2: DownloadRecord לא מאחסן יותר מפתח cache או הפניית blob. Chrome Cache API אינו בשימוש. כל הגנת תוכן מטופלת על ידי nginx proxy_cache במערכת הקבצים של Linux VM.

SPA תמיד זמין מקומית. ללא תלות בענן להפעלת אפליקציה. עדכון = בנייה מחדש + פריסת תמונת Docker.

הערות המשכיות

כיצד צוות המשך מרחיב זאת

  • זהות משתמש: Content API מקדים-מחווט middleware אימות אך עוקף אותו לנתיבי קצה. הוסף passport.js; החלף deviceId ב-userId ב-DeviceProfile. לא נדרשים שינויים אחרים.
  • אחסון אובייקטים (S3/MinIO): ContentFileService חושף store / retrieve / delete. החלף את מימוש מערכת הקבצים — שכבת השירות וה-API לא משתנים.
  • CDN: ה-proxy כבר מגיש עם כותרות Cache-Control ו-ETag. שילוב CDN הוא שינוי תצורה, לא שינוי קוד.
  • Push sync (WebSocket/SSE): שרת Express תומך ב-WS לצד REST. עצב ממשק SyncProvider כדי שניתן יהיה להחליף Pull ב-Push ללא שכתוב SPA.
  • אנליטיקה: LocalActions שומר deviceId + contentId + timestamp. סנכרון אצווה לשרת כשיש זהות. סכמת IndexedDB מוכנה — לא נדרשים שינויים.
  • טעינה מראש יזומה לקצה: הוסף עבודת רקע ל-proxy שמושך את כל התוכן החדש ל-cache בשינוי קטלוג. כרגע מטמין בגישה ראשונה; טעינה מראש יזומה מכסה את כל התוכן לפני בקשת משתמש ראשונה.
