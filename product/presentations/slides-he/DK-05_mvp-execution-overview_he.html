זרימה C — העלאת מנהל

כניסה מאובטחת למנהל · העלאת MP4 או PDF עם מטא-דאטה · תיוג קטגוריה ועניין · הקצה מסתנכרן תוך 30 שניות

הגדרת סיום — 12 קריטריוני קבלה

  • זרימת ההדגמה מקצה לקצה מושלמת (כל 15 השלבים)
  • הפעלה אוטומטית וגלילה של ריילס עובדים בצורה חלקה על הטאבלט
  • הפעלת וידאו מתחילה ≤ 2 שניות ברשת סבירה
  • חיפוש ספרייה מחזיר תוצאות תוך 5 שניות
  • PDF מתרנדר נכון בדפדפן ל-3+ מסמכי בדיקה
  • וידאו שהורד מופעל אופליין ללא רשת
  • PDF שהורד מתרנדר אופליין ללא רשת
  • העלאת מנהל → נראות בקצה ≤ 30 שניות לאחר סנכרון
  • בחירת תחומי עניין מסננת את פיד הריילס נכון
  • 15 פריטי הדגמה (10 קבצי PDF + 5 סרטונים) טעונים וניתנים לניווט
  • כל האינטראקציה בתוך Chrome — ללא חלונות קופצים, ללא אפליקציות חיצוניות
  • האפליקציה מטפלת בנפילת רשת בצורה חינה — ללא קריסות

צוות

הצוות

Dev A — שרת ענן

ראשי: שרת API (Node.js + Express), סכמת PostgreSQL ומיגרציות, טיפול בקבצים, אימות מנהל
משני: פריסת ענן, תצורת nginx TLS, בדיקות עומס

Dev B — Edge SPA ופרוקסי

ראשי: Edge SPA (HTML + TypeScript + Vite), פיד ריילס, נגן וידאו, שכבת אופליין IndexedDB, זרימת הורדה
משני: הגדרת Docker לפרוקסי הקצה

Dev C — Admin SPA וספרייה

ראשי: Admin SPA (HTML + TypeScript + Vite), טופס העלאה, רשימת תוכן, CRUD קטגוריה/עניין
משני: תצוגת ספרייה, מציג PDF, טעינת תוכן הדגמה

Sprint 1 · שבועות 1–2

תשתית — כולם לא חסומים

Dev A — תשתית שרת

הגדרת Monorepo (pnpm workspaces) · סוגי TypeScript משותפים · סכמת PostgreSQL (מיגרציה 001) · שלד שרת Express · אימות מנהל (JWT) · נקודות קצה של העלאה ורשימה

Dev B — שלד Edge SPA

HTML + TypeScript + Vite ב-packages/client · פס לשוניות 3 (Reels / ספרייה / הורדות) · ניתוב (History API) · CSS בסיסי גלובלי · הגדרת IndexedDB · אתחול DeviceProfile

Dev C — Admin SPA + Edge Proxy

HTML + TypeScript + Vite ב-packages/admin · מסך כניסה מחובר לנקודת קצה JWT · Dockerfile + nginx.conf לפרוקסי קצה · פרוקסי מתחיל ומגיש על localhost:8080

קריטריוני יציאה Sprint 1

  • pnpm install && pnpm build מצליח בכל החבילות
  • VM הענן כולל PostgreSQL פועל; מיגרציה 001 הוחלה
  • GET /api/health מחזיר 200
  • כניסת מנהל מחזירה אסימון JWT
  • מיכל Docker פרוקסי קצה מתחיל ומיישר GET /api/health מהענן
  • מסך כניסה Admin SPA מגיע לשרת
  • Edge SPA נטען ב-Chrome דרך localhost:8080

אבן דרך M1: סוף שבוע 2 — Monorepo בונה; סכמת DB קיימת; כל 3 האפליקציות מתחילות; פרוקסי קצה מיישר בהצלחה

Sprint 2 · שבועות 3–4

פייפליין תוכן — העלאה לקצה

Dev A

GET /api/catalog (תגובת קטלוג מלאה) · GET /api/content/:id/file עם תמיכה בבקשת range (206) · נקודת קצה של תמונה ממוזערת · CRUD מלא לקטגוריות ותחומי עניין · החלפת קובץ + קפיצת גרסה

Dev B

סנכרון קטלוג Edge SPA → IndexedDB · רשימה שטוחה של ספרייה עם כרטיסי תוכן · נגן וידאו בדפדפן (HTML5) · מציג PDF דרך PDF.js · כפתור רענון ידני

Dev C

טופס העלאה של מנהל (בורר קבצים, כותרת, תיאור, בוחרי קטגוריה/עניין, פס התקדמות) · רשימת תוכן מנהל (ניתנת למיון) · טופס עריכה · מחיקה עם אישור · CRUD UI לקטגוריות ועניינים · סקריפט נתוני seed

קריטריוני יציאה Sprint 2

  • העלאת וידאו דרך מנהל → מופיע ב-GET /api/catalog
  • העלאת PDF דרך מנהל → מופיע ב-GET /api/catalog
  • Edge SPA מסנכרן קטלוג → פריטים מופיעים ברשימה השטוחה
  • קובץ וידאו מוזרם עם בקשות range (תגובה 206 מאושרת)
  • PDF נפתח ומתרנדר בדפדפן מ-Edge SPA
  • מחיקת תוכן במנהל → הוסר מהקטלוג בסנכרון הבא
  • פרוקסי הקצה שומר בקובץ תוכן במטמון (X-Cache-Status: HIT בבקשה 2)

אבן דרך M2: סוף שבוע 4 — העלאה → קטלוג → זרם/צפייה על מכשיר קצה; קטגוריות ותחומי עניין קיימים

Sprint 3 · שבועות 5–6

חוויית ריילס — ספרינט הסיכון הגבוה ביותר

⚠ מדוע זהו ספרינט הסיכון הגבוה ביותר

דורש ש-scroll-snap + IntersectionObserver + מחוות מגע + מדיניות הפעלה אוטומטית של Chrome + prefetch וידאו יעבדו יחד על חומרת הטאבלט בפועל. חייב להיות מאומת בפועל עד סוף שבוע 6 (M3). אם נכשל, מושכים ידיות צמצום מיידית.

Dev B (ראשי) + Dev C (תמיכה)

פיד אנכי מסך מלא (CSS scroll-snap) · הפעלה אוטומטית דרך IntersectionObserver · מחוות החלקה (touchstart/touchend) · שכבת tap-to-start לוידאו ראשון · UI שכבה (כותרת, לייק, שמירה, הורדה) · prefetch וידאו ב-30% הפעלה · סינון לפי תחומי עניין · מסך בחירת עניין · מצב פיד ריק

Dev C

עץ קטגוריות ספרייה (2 רמות, ניתן לכיווץ) · תגי סוג תוכן · שורת חיפוש טקסט (סינון בצד הלקוח) · כפתור הורדה על כרטיסי ספרייה · הודעות מצב ריק

קריטריוני יציאה Sprint 3

  • פיד הריילס גולל בחלקות על טאבלט 10″ (ללא עיכוב; אושר בפועל)
  • הוידאו הראשון מתחיל תוך ≤ 2 שניות ברשת סבירה
  • הפעלה אוטומטית עובדת לאחר מחווה ראשונה של המשתמש
  • גלילה למעלה → וידאו הבא; גלילה למטה → וידאו קודם
  • בחירת עניין: בחר 2 עניינים → פיד מציג רק סרטונים תואמים
  • עץ קטגוריות ספרייה: הקש קטגוריה → רשימה מסוננת
  • חיפוש טקסט: הקלד מונח → רשימה מסוננת תוך 1 שניה
  • prefetch וידאו: גלילה לוידאו הבא מתחילה מהר יותר (איכותי)

אבן דרך M3: סוף שבוע 6 — פיד בסגנון TikTok מאומת בפועל על טאבלט 10″. אם M3 בסיכון → מושכים ידיות צמצום מיידית.

Sprint 4 · שבועות 7–8

ספרייה, הורדות & אופליין

Dev B

זרימת הורדה (fetch → מטמון פרוקסי → IndexedDB DownloadRecord) · פס התקדמות הורדה · לשונית הורדות (רשימה עם כותרת, סוג, גודל, תאריך) · הפעלה/צפייה בתוכן שהורד דרך מטמון פרוקסי · מחיקה מהורדות · מחוון מצב רשת (polling ל-/api/health) · הודעות אופליין בריילס ובספרייה · ניקוי עזובים בסנכרון

Dev C

תג "עודכן": השווה גרסת DownloadRecord עם גרסת קטלוג · מצב כפתור הורדה (הורד vs. הורדה) · ניווט עמודי PDF (הקודם/הבא, מונה) · תצוגת נגן וידאו עצמאית · עקביות תצוגת פרטי תוכן בין ריילס / ספרייה / הורדות

Dev A

בדיקת אופליין Nginx: עצור שרת ענן → אשר שה-SPA נטען, הקטלוג מגיש מטמון עמום, קבצים מוגשים ממטמון · תיעד תצורת nginx להגשת עמום · בדיקת עומס: 5 זרמי וידאו במקביל; עבור ל-nginx X-Accel-Redirect אם נדרש · אמת כותרות Accept-Ranges, Content-Length, ETag

קריטריוני יציאה Sprint 4

  • הורד וידאו → סמל "הורד" → מופיע בלשונית הורדות
  • הורד PDF → מופיע בלשונית הורדות
  • נתק רשת → לשונית הורדות פועלת במלואה
  • אופליין: הפעל וידאו שהורד → מופעל ללא רשת
  • אופליין: פתח PDF שהורד → מתרנדר ללא רשת
  • מחוון אופליין מופיע בריילס ובספרייה כשמנותק
  • תג "עודכן" מופיע לאחר שמנהל מעדכן פריט שהורד
  • מחק פריט שהורד → הוסר מלשונית הורדות
  • ניקוי עזובים: מחק תוכן במנהל → סנכרון → פריט הוסר

אבן דרך M4: סוף שבוע 8 — הורד וידאו + PDF → נתק → הפעל/צפה אופליין; תג "עודכן" עובד

Sprint 5 · שבועות 9–10

פורטל ניהול + אינטגרציה מלאה

Dev A — מעבר אינטגרציית API

בדיקה מקצה לקצה של כל 21 נקודות קצה API · תיקון פערי אימות (סוג MIME, בדיקת magic-byte, גודל קובץ, עומק קטגוריה) · אימות משך וידאו (< 180 שניות) · ביקורת תגובות שגיאה · VM ענן פרוסה עם HTTPS

Dev C — ליטוש סופי פורטל ניהול

CRUD קטגוריה: סידור מחדש, UI הגבלת 2 רמות · CRUD עניין: ליטוש סופי · רשימת תוכן: מיון וסינון לפי סוג · טופס העלאה: אימות בצד לקוח, הודעות שגיאה ברורות · זרימת עדכון תוכן

Dev B — הליכת הדגמה

הליכת סקריפט הדגמה מלא של 15 שלבים על המכשיר בפועל · תיקון בעיות UX שנמצאו · מדידת זמן התחלת הפעלת וידאו · כיוון prefetch אם נדרש

קריטריוני יציאה Sprint 5

  • כל 21 נקודות קצה API מגיבות נכון (בדיקת אינטגרציה ידנית)
  • סקריפט הדגמה מלא רץ מקצה לקצה ללא שגיאות ב-staging
  • AC-1 עד AC-12 מתועדים; כל קריטריון "חובה" ירוק או מסולם
  • VM ענן פרוסה ונגישה דרך HTTPS
  • דימוי Docker של פרוקסי קצה בנוי ורץ על מכשיר הקצה

אבן דרך M5: סוף שבוע 10 — כל 15 שלבי הדגמה רצים מקצה לקצה; כל AC עובר או מסולם לבעל עניין

Sprint 6 · שבועות 11–12

ייצוב & הכנת הדגמה

Dev C — תוכן הדגמה (שבוע 11)

טעינת כל 15 פריטי ההדגמה: 10 קבצי PDF + 5 סרטונים · הקצאת קטגוריות ותחומי עניין לכל פריט · אימות כל פריט בקטלוג, ספרייה וריילס · אימות תמונות ממוזערות

Dev B — הגדרת מכשיר (שבוע 11)

כיוון ביצועים: זמן התחלת וידאו, חלקות גלילה, תגובתיות החלקה · מכשיר קצה: WSL2/VirtualBox + Docker + מיכל פרוקסי קצה (restart always) · מצב קיוסק Chrome: ללא חלונות קופצים, ללא אינטראקציות OS, ניווט נשאר בתוך האפליקציה

Dev A — רגרסיה ויציבות (שבוע 11)

בדיקת רגרסיה: כל נקודות קצה API לאחר טעינת תוכן הדגמה · VM ענן: pm2/systemd ל-Node.js; nginx TLS; גיבוי PostgreSQL · תיקון רגרסיות

קריטריוני יציאה Sprint 6

  • 15 פריטי הדגמה טעונים וניתנים לניווט (AC-10)
  • כל 12 קריטריוני הקבלה (AC-1 עד AC-12) עוברים על מכשיר ההדגמה בפועל
  • חזרה על הדגמה הושלמה פעמיים ללא בעיות חוסמות
  • מצב קיוסק Chrome אושר: אפס אפליקציות חיצוניות, חלונות קופצים, או אינטראקציות OS (AC-11)
  • מיכל Docker פרוקסי קצה מאתחל אוטומטית לאחר הפעלת מכשיר מחדש

אבן דרך M6: סוף שבוע 12 — 15 פריטים טעונים; חזרות פעמיים; כל AC ירוק; מכשיר מוגדר ויציב. הדגמה מוכנה.

לאחר MVP

מפת דרך להמשך — לאחר MVP

1. Delta Sync עדיפות 1

החלף משיכת קטלוג מלאה בתגובה מבוססת delta באמצעות ?since={timestamp}. השרת מחזיר רק פריטים שהשתנו + מערך נמחקים. שיפור המדרגיות בעדיפות הגבוהה ביותר ככל שהקטלוג גדל.

2. אימות משתמש

ל-Edge SPA אין זהות משתמש ב-MVP. כאשר אימות מתווסף, DeviceProfile.deviceId הופך ל-userId; API מוסיף middleware אימות; LocalAction מקבל דגל synced להעלאה אצווה.

3. HLS Adaptive Streaming

MP4 + בקשות range עובד לקטעים ≤ 3 דקות. לתוכן ארוך יותר או רוחב פס משתנה, הוסף HLS segmentation (ffmpeg בהעלאה) ונגן HLS (hls.js). אלמנט <video> ניתן להחלפה ללא שינוי שאר הארכיטקטורה.

4. Prefetch יזום בקצה

הוסף משרה מתוזמנת בתוך מיכל Docker (cron) שמושכת את כל התוכן למטמון לפי לוח זמנים — במקום להמתין לבקשת משתמש ראשונה. מבטיח שכל התוכן זמין אופליין אפילו לפני גישה ראשונה.

5. כיסוי בדיקות

ל-MVP אין בדיקות אוטומטיות. הצוות הבא צריך להוסיף בדיקות יחידה לשכבת הסירוויס (CatalogService, ContentFileService, AuthService) ובדיקות אינטגרציה לנקודות קצה קריטיות לפני הוספת תכונות חדשות.

6. ניטור

הוסף רישום בקשות מובנה בצד שרת (pino) והתראות שגיאה (Sentry) לפני כניסה לייצור עם משתמשים אמיתיים. הגדר לוח Grafana או שווה ערך ללאטנסי ושיעור שגיאות.
