## 1. מטרת המסמך

מסמך זה הוא תוכנית הביצוע של TactiTok MVP. הוא עונה על:

- כיצד 12 שבועות של עבודה מאורגנים בין 3 מפתחים סטודנטים
- מה נבנה באיזה סדר, ומדוע
- מהם אבני הדרך ונקודות בדיקת הדמו
- מהם המסלול הקריטי והתלויות הקשות
- מתי וכיצד להשתמש במנופי צמצום היקף אם הזמן קצר
- מה פירוש "גמור" בכל שלב

זהו המסמך האחרון בסט המפרטים. כל המסמכים הקודמים (תקציר מוצר → חוזה API) מזינים אותו. הארכיטקטורה, מודל הנתונים וחוזה ה-API קבועים. תוכנית זו מממשת אותם.

## 4. תוכניות ספרינט

### ספרינט 1 — יסודות (שבועות 1–2)

**מטרה:** לכולם יש סביבת פיתוח עובדת. שלושת האפליקציות ניתנות להפעלה. סכמת מסד הנתונים קיימת. ה-edge proxy פועל מקומית.

#### שבוע 1

**Dev A — יסודות Backend:**
- הגדרת monorepo (`pnpm workspaces`): `server`, `shared`, `client`, `admin`, `edge-proxy`
- הגדרה ופרסום סוגי TypeScript משותפים (`packages/shared/src/types.ts`) — `ContentItemDTO`, `CatalogResponse`, `CategoryDTO`, `InterestDTO`, `LoginRequest`, `LoginResponse`, `ApiError`
- יצירת סכמת PostgreSQL (מיגרציה 001): `content_items`, `categories`, `interests`, `content_categories`, `content_interests`
- שלד שרת Express: מבנה ניתוב, handler שגיאות, רישום בקשות
- Endpoint בדיקת תקינות: `GET /api/health → 200 { status: 'ok' }`

**Dev B — שלד Edge SPA:**
- הגדרת React + TypeScript + Vite ב-`packages/client`
- סרגל טאבים תחתון עם 3 טאבים: ריילס, ספרייה, הורדות
- מסך placeholder לכל טאב
- ניתוב (React Router או TanStack Router)
- בסיס CSS גלובלי: פריסת מסך מלא, מותאמת לטאבלט

**Dev C — שלד Admin SPA + edge proxy:**
- הגדרת React + TypeScript + Vite ב-`packages/admin`
- מסך כניסה עם שדה סיסמה (לא מחובר לשום דבר עדיין)
- מסך dashboard placeholder
- Dockerfile + `nginx.conf` עבור edge proxy (`packages/edge-proxy`):
  - הגשת קבצי SPA סטטיים מ-`/var/www/spa/`
  - proxy ל-`/api/*` לשרת ענן
  - הגדרת `proxy_cache` בסיסית
- אימות שקונטיינר Docker מופעל ומגיש דף HTML בדיקה על `localhost:8080`

#### שבוע 2

**Dev A:**
- אימות ניהולי: `POST /api/admin/login` + `POST /api/admin/logout` עם JWT
- Middleware אימות לכל נתיבי `/api/admin/*`
- `POST /api/admin/content` (endpoint העלאה): Multer, אימות MIME, אחסון filesystem, הכנסה ל-DB
- `GET /api/admin/content` (endpoint רשימה)

**Dev B:**
- חיבור edge SPA ל-edge proxy: כל קריאות ה-API הולכות ל-`localhost:8080`
- הגדרת IndexedDB: store ‏`deviceProfile`, store ‏`catalogCache`, store ‏`downloads`, store ‏`localActions`
- אתחול DeviceProfile: יצירת `deviceId` בפתיחה הראשונה
- מסך בחירת עניינים (ממשק בלבד — אין עניינים ב-DB עדיין; הצגת placeholder)

**Dev C:**
- חיבור כניסת Admin SPA ל-`POST /api/admin/login`; שמירת JWT ב-`sessionStorage`
- Dashboard ניהולי: קריאה ל-`GET /api/admin/content`; הצגת רשימת תוכן placeholder
- מסכי קטגוריות ועניינים ניהוליים (stub — לוגיקת CRUD בספרינט 5)

**קריטריוני יציאה מספרינט 1:**
- [ ] ‏`pnpm install && pnpm build` מצליח בכל החבילות
- [ ] ב-VM ענן פועל PostgreSQL; מיגרציה 001 מיושמת
- [ ] ‏`GET /api/health` מחזיר 200
- [ ] כניסת אדמין מחזירה טוקן JWT
- [ ] קונטיינר Docker ל-edge proxy מופעל ומעביר proxy ל-`GET /api/health` מהענן
- [ ] מסך כניסה Admin SPA מגיע לשרת
- [ ] Edge SPA נטען ב-Chrome דרך `localhost:8080`

### ספרינט 2 — צינור תוכן (שבועות 3–4)

**מטרה:** ניתן להעלות קובץ דרך פורטל הניהול, להופיע בקטלוג, ולהזרים/להוריד במכשיר הקצה. זהו עמוד השדרה שכל שאר התכונות מצטרפות אליו.

#### שבוע 3

**Dev A:**
- `GET /api/catalog` — תגובת קטלוג מלאה (פריטים + קטגוריות + עניינים) עם סריאליזציה תקינה
- `GET /api/content/:id/file` — הזרמת קובץ עם תמיכת בקשת range (`Accept-Ranges`, `Content-Range`, `206`)
- `GET /api/content/:id/thumbnail` — הגשת תמונה ממוזערת (או 404)
- `PUT /api/admin/content/:id` — עדכון מטה-נתונים
- `DELETE /api/admin/content/:id` — מחיקה קשה (קובץ + רשומת DB)
- `GET /api/admin/content/:id` — פריט יחיד

**Dev B:**
- סנכרון קטלוג Edge SPA: `GET /api/catalog` → ניתוח → כתיבה ל-IndexedDB ‏`catalogCache`
- תצוגת חותמת "סונכרן לאחרונה"
- כפתור ריענון ידני (מפעיל סנכרון מחדש)
- תצוגת ספרייה: רשימה שטוחה של כל פריטי התוכן (ללא קטגוריות עדיין; רשימה בלבד)
- כרטיס פריט תוכן בסיסי: כותרת, אייקון סוג, גודל קובץ

**Dev C:**
- טופס העלאה ניהולי: בוחר קבצים, כותרת, תיאור, בוחר קטגוריה (בחירה מרובה), בוחר עניין (בחירה מרובה), כפתור העלאה
- סרגל התקדמות העלאה (אירוע `progress` של XMLHttpRequest או Fetch + ReadableStream)
- רשימת תוכן ניהולית: טבלה ניתנת למיון (כותרת, סוג, תאריך, פעולות)
- טופס עריכה ניהולי (עדכון מטה-נתונים)
- מחיקה ניהולית עם תיבת דו-שיח לאישור

#### שבוע 4

**Dev A:**
- `POST /api/admin/categories`, `PUT`, `DELETE` (CRUD מלא)
- `GET /api/admin/categories`
- `POST /api/admin/interests`, `PUT`, `DELETE` (CRUD מלא)
- `GET /api/admin/interests`
- `PUT /api/admin/content/:id/file` — החלפת קובץ, הגדלת גרסה
- `PUT /api/admin/content/:id/thumbnail` — העלאת תמונה ממוזערת

**Dev B:**
- נגן וידאו בדפדפן (אלמנט HTML5 `<video>`): הפעלה/השהיה, סרגל התקדמות, השתקה
- פתיחת וידאו מספרייה → תצוגת נגן
- מציג PDF: `fetch()` קובץ ה-PDF → העברת `ArrayBuffer` ל-`PDF.js`; ניווט עמוד-עמוד
- פתיחת PDF מספרייה → מציג PDF

**Dev C:**
- ממשק CRUD קטגוריות ניהולי: תצוגת עץ, הוספת רמה עליונה, הוספת ילד, שינוי שם, מחיקה
- ממשק CRUD עניינים ניהולי: רשימה, הוספה, שינוי שם, מחיקה
- העלאת תמונה ממוזערת ניהולית (כפתור נפרד בטופס עריכה)
- סקריפט נתוני seed: 3 קטגוריות, 3 תת-קטגוריות, 3 עניינים (לשימוש פיתוח)

**קריטריוני יציאה מספרינט 2:**
- [ ] העלאת וידאו דרך ניהול → מופיע ב-`GET /api/catalog`
- [ ] העלאת PDF דרך ניהול → מופיע ב-`GET /api/catalog`
- [ ] Edge SPA מסנכרן קטלוג → פריטים מופיעים ברשימה שטוחה בספרייה
- [ ] קובץ וידאו מוזרם עם בקשות range (אושרה תגובת 206)
- [ ] PDF נפתח ומרונדר בדפדפן מ-Edge SPA
- [ ] מחיקת תוכן בניהול → מוסר מקטלוג בסנכרון הבא
- [ ] Edge proxy שומר קובץ תוכן במטמון (אישור `X-Cache-Status: HIT` בבקשה שנייה)

### ספרינט 3 — חוויית ריילס (שבועות 5–6)

**מטרה:** הפיד בסגנון TikTok עובד על טאבלט. הפעלה אוטומטית, החלקה, טעינה מוקדמת וסינון עניינים כולם עובדים. זהו היכולת בסיכון הגבוה ביותר — יש לאמת אותה על החומרה בפועל עד סוף שבוע 6.

#### שבוע 5

**Dev B (ראשי) + Dev C (תמיכה):**
- פיד ריילס אנכי מסך מלא תוך שימוש ב-CSS ‏`scroll-snap`
- הפעלה אוטומטית של וידאו כשגוללים לתצוגה (`IntersectionObserver`)
- זיהוי מחוות החלקה למעלה / למטה (אירועי `touchstart`/`touchend` מובנים; גיבוי: אירוע `wheel` לבדיקת שולחן עבודה)
- וידאו ראשון: טיפול במדיניות הפעלה אוטומטית של Chrome — שימוש בשכבת-על "הקש להתחיל" עבור הוידאו הראשון; וידאו עוקבים מופעלים אוטומטית
- ממשק שכבת-על וידאו: כותרת, תיאור (קיצוץ 2-שורות), כפתור לייק, כפתור שמירה, כפתור הורדה
- כפתור לייק: מחליף `LocalAction` ב-IndexedDB; מעדכן מצב אייקון
- כפתור שמירה: מחליף `LocalAction` ב-IndexedDB; מעדכן מצב אייקון
- מחוון טעינה/חיץ (מוצג בזמן טעינת וידאו; מוסתר באירוע `canplay`)

**Dev A:**
- חקירה ואישור: ‏`proxy_cache_valid 200 206 30d` שומר כראוי תגובות בקשת range ב-nginx
- אם שמירת בקשות range במטמון שבורה: מימוש פתרון עקיף (חימום מוקדם של מטמון עם GET מלא לפני טעינת וידאו; תיעוד ב-`notes/decisions.md`)

**בדיקה על מכשיר בסוף שבוע 5:**
- טעינת Docker edge proxy על VM Linux; Chrome על Windows → `localhost:8080`
- העלאת וידאו בדיקה אחד, סנכרון קטלוג, פתיחת פיד ריילס
- אימות: הפעלה אוטומטית עובדת, החלקה עובדת, וידאו חלק

#### שבוע 6

**Dev B:**
- טעינה מוקדמת של וידאו: כשהוידאו הנוכחי מגיע ל-30% הפעלה, שליחת בקשת `Range: bytes=0-{N}` לוידאו הבא בפיד. אחסון התגובה החלקית בזיכרון. כשהמשתמש מחליק, התחלת `<video>` מהbytes שנטענו מראש.
- סינון פיד מבוסס עניינים: קריאת `DeviceProfile.selectedInterestIds` מ-IndexedDB → סינון `CachedCatalog.items` כאשר `interestIds` מצטלב
- מסך בחירת עניינים: מוצג בפתיחה הראשונה (כאשר `DeviceProfile.selectedInterestIds` ריק); מציג את כל העניינים כלחצני chip; הקשה על "התחל" שומרת ומנווטת לריילס
- גישה להגדרות עניינים מאזור פרופיל/הגדרות (פותח מחדש את בחירת עניינים)
- מצב ריק של פיד: הודעת "אין וידאואים התואמים לעניינים שלך" עם קישור להגדרות
- השפלת רשת חינה: אם וידאו נעצר, הצגת מחוון חיץ; לא להתרסק

**Dev C:**
- עץ קטגוריות ספרייה: עץ 2-רמות ניתן לכיווץ בפאנל שמאל / חלק עליון
- הקשה על קטגוריה מסננת את רשימת התוכן
- אייקון סוג תוכן (תג וידאו לעומת PDF)
- שדה חיפוש טקסט: מסנן `CachedCatalog.items` בצד הלקוח על `title + description` בזמן שהמשתמש מקליד
- הודעות מצב ריק: "אין פריטים בקטגוריה זו" / "לא נמצאו תוצאות"
- כפתור הורדה על כרטיסי פריטי תוכן בספרייה

**קריטריוני יציאה מספרינט 3:**
- [ ] פיד ריילס גולל בחלקות על טאבלט 10 אינץ' (ללא ניגוד; אושר ידנית)
- [ ] הוידאו הראשון מתחיל תוך ≤ 2 שניות ברשת סבירה
- [ ] הפעלה אוטומטית עובדת לאחר מחווה משתמש ראשונה
- [ ] החלקה למעלה → וידאו הבא; החלקה למטה → וידאו קודם
- [ ] בחירת עניינים: בחירת 2 עניינים → הפיד מציג רק וידאואים תואמים
- [ ] עץ קטגוריות ספרייה: הקשה על קטגוריה → רשימה מסוננת
- [ ] חיפוש טקסט: הקלדת מונח → רשימה מסוננת תוך שנייה אחת
- [ ] טעינה מוקדמת של וידאו: החלקה לוידאו הבא; ההפעלה מתחילה מהר יותר מאשר בלי טעינה מוקדמת (איכותי)

### ספרינט 4 — ספרייה, הורדות ואופליין (שבועות 7–8)

**מטרה:** משתמשים יכולים להוריד וידאואים ו-PDFs, לצפות בהם אופליין, ולנהל את ההורדות שלהם. מצב אופליין עובד בשקיפות.

#### שבוע 7

**Dev B:**
- זרימת הורדה: המשתמש מקיש על הורדה → `fetch("/api/content/{id}/file")` מ-proxy → קריאת תגובה עד השלמה (מאלצת את ה-proxy לשמור את הקובץ המלא במטמון) → כתיבת `DownloadRecord` ל-IndexedDB → עדכון טאב הורדות
- התקדמות הורדה: מעקב אחר bytes שהתקבלו בזמן fetch; עדכון סרגל התקדמות בכפתור ההורדה
- טאב הורדות: רישום כל רשומות `DownloadRecord` מ-IndexedDB; הצגת כותרת, סוג, גודל, תאריך הורדה
- הפעלה/צפייה בוידאו שהורד: אותו `<video src="/api/content/{id}/file">` → proxy מגיש ממטמון (אין נתיב קוד מיוחד)
- הפעלה/צפייה ב-PDF שהורד: אותו `fetch("/api/content/{id}/file")` → proxy מגיש ממטמון → PDF.js מרנדר
- מחיקת פריט שהורד: הסרת `DownloadRecord` מ-IndexedDB; מטמון proxy עשוי להחזיק עדיין קובץ (ליידע משתמש שה-proxy מנהל את הקובץ)

**Dev C:**
- לוגיקת תג "מעודכן": בסנכרון קטלוג, השוואת `DownloadRecord.version` עם `CachedCatalog.items[].version`; אם גרסת קטלוג גבוהה יותר, סימון פריט בטאב הורדות עם תג "מעודכן"
- תג "מעודכן" בספרייה: אם `updatedAt` של פריט חדש יותר מ-`CachedCatalog.lastSyncedAt` מהסנכרון הקודם (היוריסטיקה משוערת) — חלופה פשוטה יותר: הצגת תג רק על פריטים שהורדו שבהם אי-התאמת גרסה ברורה
- מצב כפתור הורדה: אם פריט נמצא ב-IndexedDB של `DownloadRecord`, הצגת אייקון "הורד"; אם לא, הצגת כפתור "הורדה"

**Dev A:**
- בדיקת אופליין של edge proxy nginx: סימולציה של אי-נגישות ענן (`iptables` או עצירת שרת הענן); אישור:
  - Edge SPA נטען (מוגש מתמונת Docker)
  - ‏`GET /api/catalog` מגיש מטמון ישן
  - ‏`GET /api/content/:id/file` מגיש קובץ ממטמון
- תיעוד כל שינויי הגדרת nginx הנדרשים להגשת stale ב-`packages/edge-proxy/nginx.conf`

#### שבוע 8

**Dev B:**
- מחוון סטטוס רשת: סקר `GET /api/health` דרך edge proxy; אם נכשל או מחזיר תגובת stale, הצגת מחוון "אופליין" בכותרת
- זיהוי header תגובת `X-Cache-Status: STALE`: אם proxy מחזיר קטלוג ישן, הצגת "סונכרן לאחרונה {חותמת זמן}" במקום "סונכרן עכשיו"
- ניקוי עזובים בסנכרון: לאחר סנכרון קטלוג מוצלח, הסרת רשומות `DownloadRecord` ו-`LocalAction` עבור IDs תוכן שאינם עוד בקטלוג
- מחוון אופליין בתצוגת ריילס: אם אופליין והוידאו לא במטמון, הצגת הודעת "לא זמין אופליין"
- מצב ריק של טאב הורדות: "עדיין אין הורדות — הקש על כפתור ההורדה על כל וידאו או מסמך"

**Dev C:**
- ניווט עמודי PDF: כפתורי הקודם/הבא, מונה עמודים (לדוג', "3 / 12")
- תצוגת נגן וידאו עצמאי (נפתח מספרייה או הורדות): נגן מסך מלא עם כפתור חזרה; הפעלה/השהיה, סרגל התקדמות, השתקה; כפתור הורדה
- עקביות תצוגת פרטי תוכן: אותם רכיבי נגן/מציג משמשים בריילס, ספרייה והורדות

**Dev A:**
- בדיקת עומס: סימולציה של 5 בקשות וידאו stream מקביליות ל-Node.js; אישור ללא צוואר בקבוק; אם נדרש, מעבר ל-nginx ‏`X-Accel-Redirect` להגשת קבצים
- הבטחת נכונות headers ‏`Accept-Ranges`, `Content-Length` ו-`ETag` לכל תגובות קובץ תוכן

**קריטריוני יציאה מספרינט 4:**
- [ ] הורדת וידאו → כפתור הורדה מציג "הורד" → פריט מופיע בטאב הורדות

### ספרינט 5 — השלמת פורטל הניהול + אינטגרציה (שבועות 9–10)

**Dev C:**

**Dev B:**

- ביצועים: מדידת זמן התחלת הפעלת וידאו; כוונון תזמון טעינה מוקדמת אם נדרש

## 6. מפת תלויות (מסלול קריטי)

התלויות הבאות הן קשות: משימה במורד הזרם לא יכולה להתחיל עד שהמשימה במעלה הזרם הושלמה.

```
shared/types.ts (Dev A, שבוע 1)
    ├── נתיבי packages/server תלויים ב-DTOs
    ├── קריאות API של packages/client תלויות ב-DTOs
    └── קריאות API של packages/admin תלויות ב-DTOs

סכמת PostgreSQL (Dev A, שבוע 1)
    └── כל ה-endpoints של השרת תלויים בסכמה

GET /api/catalog (Dev A, שבוע 3)
    └── סנכרון קטלוג Edge SPA (Dev B, שבוע 3)
        ├── סינון תצוגת ספרייה לפי קטגוריה (Dev C, שבוע 6)
        └── סינון פיד מבוסס עניינים (Dev B, שבוע 6)

GET /api/content/:id/file (Dev A, שבוע 3)
    ├── נגן וידאו (Dev B, שבוע 4)
    ├── מציג PDF (Dev B/C, שבוע 4)
    ├── זרימת הורדה (Dev B, שבוע 7)
    └── טעינה מוקדמת של ריילס (Dev B, שבוע 6)

Edge proxy Docker פועל (Dev C → Dev B, שבוע 1)
    └── כל תכונות הקצה תלויות ב-proxy נגיש

בדיקת proxy אופליין (Dev A, שבוע 7)
    └── אישור הפעלה אופליין (Dev B, שבוע 8)
```

**מסלול קריטי (השרשרת הארוכה ביותר):**

```
הגדרת monorepo → סוגים משותפים → סכמה → endpoint קטלוג → סנכרון קטלוג
    → סינון עניינים → פיד ריילס → בדיקת ריילס על מכשיר (M3, שבוע 6)
```

**אימות ריילס ב-M3 הוא אבן הדרך בסיכון הגבוה ביותר.** אם נכשל, יש עדיין 6 שבועות להתאוששות או לצמצום היקף.

## 8. תוכנית תגובה לסיכונים (מנופי צמצום היקף)

עץ ההחלטה הבא מנחה לצמצום היקף. יש למשוך מנופים לפי סדר עדיפות. לא לחכות עד שבוע 12 — אם ספרינט מתעכב, למשוך מנופים בסוף אותו ספרינט.

### טריגר: ספרינט מתעכב > 2 ימים עד שבוע 5

- ביטול תג "מעודכן" (CAP-3.7): חוסך כ-2–3 ימים
- השפעה: משתמשים לא רואים תג כשתוכן מתעדכן; הם עדיין מקבלים תוכן עדכני לאחר סנכרון

### טריגר: פיד ריילס לא עובד על מכשיר עד סוף שבוע 6 (M3 בסיכון)

משיכת **מנוף 2**:
- ביטול כפתורי לייק/שמירה (CAP-2.7, CAP-2.8): חוסך כ-2–3 ימים
- השפעה: אין סימניות; משתמשים מסתמכים על טאב הורדות בלבד

### טריגר: כל תכונת ספרינט 4 לא עובדת עד סוף שבוע 8

משיכת **מנוף 3**:
- ביטול מסך בחירת עניינים + סינון פיד (CAP-1.4, CAP-1.5, CAP-2.5): חוסך כ-4–5 ימים

## 14. הערות להמשך

הנחיות לצוות המשך שיאסוף לאחר ה-MVP:

- **עדיפות שבוע 1 לצוות הבא:** קריאת כל 7 מסמכי המפרט לפי סדר; `notes/decisions.md` לכל ההחלטות הנעולות; `notes/open-questions.md` לפריטים לא פתורים.
- **מיגרציות מסד נתונים:** כל שינויי סכמה חייבים להיות קבצי מיגרציה ממוספרים (Prisma או Knex). היסטוריית המיגרציה מספרינט 1 ואילך היא הרשומה הסמכותית של אבולוציית הסכמה.
- **כיסוי בדיקות:** ל-MVP אין בדיקות אוטומטיות. הצוות הבא צריך להוסיף בדיקות יחידה לשכבת השירות (CatalogService, ContentFileService, AuthService) ובדיקות אינטגרציה עבור ה-endpoints הקריטיים לפני הוספת תכונות חדשות.
- **ניטור:** הוספת רישום בקשות מובנה בצד השרת (לדוג', `pino`) והגדרת התראות שגיאה (לדוג', Sentry) לפני מעבר לייצור עם משתמשים אמיתיים.
- **אימות משתמשים:** ל-Edge SPA אין זהות משתמש. כאשר מוסיפים אימות, `DeviceProfile.deviceId` הופך ל-`userId`; ה-API מוסיף middleware אימות ל-endpoints של תוכן; רשומות LocalAction מקבלות דגל `synced` להעלאה אצווה.
- **סנכרון דלתא:** החלפת משיכה מלאה של `GET /api/catalog` בתגובה מבוססת דלתא תוך שימוש ב-`?since={timestamp}`. השרת מחזיר רק פריטים שהשתנו ומערך `deleted`. זהו שיפור המדרגיות בעדיפות הגבוהה ביותר לצמיחת קטלוג.
- **Adaptive streaming:** MP4 + בקשות range עובד עבור קטעים ≤ 3 דקות. לתוכן ארוך יותר או רוחב פס משתנה מאוד, הוספת קיטוע HLS תוך שימוש ב-ffmpeg בעת העלאה ונגן HLS (לדוג', hls.js) בלקוח. אלמנט ה-`<video>` ניתן להחלפה ללא שינוי שאר הארכיטקטורה.
- **Prefetch פרואקטיבי של Edge proxy:** הוספת משימה מתוזמנת בתוך ה-edge proxy (לדוג', cron בתוך קונטיינר Docker) שמשך את כל התוכן מהענן למטמון לפי לוח זמנים, במקום לחכות לגישת משתמש ראשונה. זה מבטיח שכל התוכן זמין אופליין גם לפני בקשת המשתמש הראשונה.
- **Docker Compose:** כאשר הגדרת מכשיר הקצה גדלה (לדוג', קונטיינרים מרובים), החלפת `docker run` בסיסי ב-`docker-compose.yml`. חבילת `packages/edge-proxy` צריכה לכלול תבנית `docker-compose.yml`.
