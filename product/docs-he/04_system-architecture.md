## 1. מטרת מסמך זה

מסמך זה מגדיר את הארכיטקטורה המינימלית הנדרשת לממש את TactiTok MVP. הוא עונה על:

- מהם רכיבי המערכת העיקריים ומה אחריותו של כל אחד
- מה רץ על מכשיר הקצה, מה רץ על השרת, ומה חיצוני
- כיצד תוכן, מטה-נתונים והורדות מקומיות מטופלים ברמה גבוהה
- כיצד המערכת מתנהגת בקישוריות לסירוגין
- מהם הממשקים העיקריים בין רכיבים
- אילו החלטות ארכיטקטורה נדחות בכוונה

כל מסמך נגזר (מודל נתונים → חוזה API → תכנית משלוח) חייב להיות עקבי עם ארכיטקטורה זו.

## 3. הגדרות / מונחים

נלקחו ממסמכים קודמים. תוספות ספציפיות לארכיטקטורה:

| מונח | הגדרה |
|------|-------|
| **שרת API** | תהליך Python FastAPI ב-VM הענן המשרת נקודות קצה REST הן לפרוקסי הקצה והן לפורטל הניהול |
| **הפשטת אחסון** | ממשק פנימי (קרא/כתוב/מחק לפי content-id) המנתק את ה-API מה-backend הפיזי של האחסון |
| **פרוקסי קצה** | פרוקסי הפוך עם מטמון nginx הרץ ב-Docker container ב-Linux VM של מכשיר הקצה; משרת את Edge SPA מקומית ומפרוקסי + שומר כל בקשות API/תוכן לשרת הענן במטמון |
| **פרוקסי תוכן** | תפקיד שרת ה-API בהגשת קבצי תוכן עם headers HTTP נכונים (range requests, cache control, CORS) |
| **חבילת SPA** | ה-HTML/JS/CSS הסטטי המרכיב את יישום הלקוח; ללא שלב בנייה — קבצים מועתקים ישירות לתמונת Docker של פרוקסי הקצה |
| **Docker container קצה** | Docker container ב-Linux VM של מכשיר הקצה שארוז nginx + קבצי Edge SPA הסטטיים; מספק הגשה מקומית ויכולת אופליין |

| Chrome קיוסק על Windows, Linux VM עם Docker על אותו מכשיר | North Star §3 + גילוי | Chrome מתחבר ל-localhost; פרוקסי קצה ב-Docker משרת SPA ושומר תוכן ענן במטמון |
| רוחב-פס לא יציב / מוגבל | North Star §3 | פרוקסי קצה שומר תוכן + מטה-נתונים מקומית; מפחית תלות בענן |

| TLS נדרש (ענן) | החלטה D13 | סיום HTTPS בשרת הענן (nginx reverse proxy) |
| MP4 מוכן-להעלאה בלבד | החלטה D14 | ללא צינור עיבוד; אמת ואחסן |
| מכשיר קצה = Windows + Linux VM מקומי | גילוי (ארכיטקטורה) | Docker container ב-Linux VM; Chrome על Windows מתחבר דרך localhost |

## 7. אחריות רכיבים

### C1 — Edge SPA

| אחריות | פרטים |
|--------|-------|
| **פיד ריילס** | גלילה אנכית בסגנון TikTok; השמעה-אוטומטית; ניווט-החלקה; סינון עניינים |
| **תצוגת ספרייה** | עץ קטגוריות 2-רמתי; חיפוש טקסט; רשימת תוכן עם מחווני סוג |
| **נגן וידאו** | אלמנט HTML5 `<video>`; פקדי נגן/עצור, סרגל התקדמות, השתק |
| **מציג PDF** | אינטגרציה PDF.js; ניווט עמוד-אחר-עמוד |
| **מנהל הורדות** | אתחול הורדות (מפעיל שמירה במטמון פרוקסי); הצגת התקדמות; ניהול פריטים שהורדו |
| **פרופיל מכשיר** | בחירת עניינים (ראשונה + הגדרות); שמירה ב-IndexedDB |
| **פעולות מקומיות** | לייק/שמירה נשמרים ב-IndexedDB; ללא סנכרון שרת ב-MVP |
| **טיפול מטה-נתונים** | בקשת קטלוג מפרוקסי קצה; שמירה ב-IndexedDB לסינון/חיפוש בצד לקוח |
| **מודעות לרשת** | זיהוי אונליין/אופליין (נגישות פרוקסי לענן); הצגת מחוון מצב; שחיקה חלקה |
| **ניווט** | סרגל לשוניות תחתי: ריילס, ספרייה, הורדות |

**מה השתנה (v0.2):** Edge SPA אינו עוד מנהל Cache API לקבצי מדיה או Service Worker לשמירת מעטפת SPA במטמון. כל שמירת תוכן במטמון מטופלת על ידי פרוקסי הקצה. ה-SPA שולח כל הבקשות ל-`localhost` (הפרוקסי), לעולם לא ישירות לענן. IndexedDB עדיין משמש לפרופיל מכשיר, פעולות מקומיות, רשומות הורדה (מטה-נתונים בלבד), וסנפשוט קטלוג מנותח לסינון בצד לקוח.

**מה השתנה (v0.3):** ה-Edge SPA הוא Vanilla HTML + CSS + JS ללא שלב קומפילציה. אין TypeScript, אין Vite build, ואין `node_modules`. קבצי ה-SPA נפתחים ישירות ב-Chrome (קצה) או מועתקים לתמונת Docker כמות שהם.

### C2 — Admin SPA

| אחריות | פרטים |
|--------|-------|
| **אימות** | מסך כניסה בסיסמה; token פעלה נשמר בזיכרון/sessionStorage |
| **ניהול תוכן** | העלאת וידאו (MP4) / PDF; מילוי טופס מטה-נתונים; עריכה; מחיקה |
| **עדכון תוכן** | החלפת קובץ לתוכן קיים; השרת מגדיל גרסה |
| **ניהול קטגוריות** | CRUD לעץ קטגוריות 2-רמתי |
| **ניהול עניינים** | CRUD לרשימת עניינים/תגיות שטוחה |
| **רשימת תוכן** | טבלה ניתנת-למיון, ניתנת-לסינון של כל פריטי התוכן |
| **אימות העלאה** | בצד לקוח: בדיקת סוג קובץ, גודל (≤100MB); השרת גם מאמת |

**מה השתנה (v0.3):** ה-Admin SPA הוא plain HTML + vanilla JS המוגש כקבצים סטטיים על ידי FastAPI ב-`/admin`. אין שלב Vite build, אין מהדר TypeScript, ואין מסגרת SPA.

### C3 — שרת API (Python FastAPI)

| אחריות | פרטים |
|--------|-------|
| **Content API** (פונה-לקצה) | `GET` מטה-נתוני קטלוג; `GET` קובץ תוכן (וידאו/PDF) עם תמיכת range request |
| **Admin API** (פונה-לניהול) | `POST` העלאה; `PUT` עדכון מטה-נתונים/קובץ; `DELETE` תוכן; CRUD קטגוריות; CRUD עניינים |
| **Auth middleware** | נקודות קצה אדמין מוגנות בפעלת סיסמה; נקודות קצה קצה הן ציבוריות |
| **הגשת קבצים סטטיים** | הגשת חבילת Admin SPA בלבד (Edge SPA נארז ב-Docker) |
| **פרוקסי תוכן** | הגשת קבצי תוכן ממאגר הקבצים עם headers נכונים (Content-Type, Accept-Ranges, Content-Length, Cache-Control) |
| **אימות** | דחיית וידאו שאינו MP4, מסמכים שאינם PDF, קבצים >100MB, וידאו >3 דקות (אם בדיקת משך מיושמת) |

**מה השתנה (v0.2):** שרת ה-API אינו עוד מגיש את חבילת Edge SPA. הוא מגיש רק את Admin SPA. מכשירי קצה מקבלים את ה-SPA מ-Docker container המקומי.

**מה השתנה (v0.3):** שרת ה-API הוא כעת Python FastAPI (מחליף את Node.js + Express). SQLAlchemy הוא ה-ORM. Alembic מנהל הגירות מסד נתונים (מחליף את Prisma migrate).

### C7 — פרוקסי קצה (nginx caching reverse proxy ב-Docker)

| אחריות | פרטים |
|--------|-------|
| **הגשת SPA** | הגשת קבצים סטטיים של Edge SPA (HTML/JS/CSS) מתמונת Docker; nginx `try_files` מטפל ב-SPA client-side routing |
| **פרוקסי API** | העברת כל בקשות `/api/*` לשרת הענן (HTTPS) |
| **שמירת תוכן במטמון** | שמירת תגובות API וקבצי תוכן (וידאו/PDF) ב-nginx `proxy_cache`; גודל מטמון ניתן-לתצורה (≥10 GB לדמו) |
| **שמירת מטה-נתונים במטמון** | שמירת תגובות API קטלוג במטמון; הגשת גרסה שמורה כשהענן אינו נגיש |
| **מצב אופליין** | כשהענן מושבת, הגשת כל התגובות השמורות באופן שקוף; Chrome/SPA לא צריך לדעת |
| **headers מטמון** | הוספת header `X-Cache-Status` (HIT/MISS/STALE) כך שה-SPA יכול לזהות רעננות |
| **לא אחראי על** | פרופיל מכשיר, פעולות מקומיות, מעקב הורדות (אלו נשארים ב-IndexedDB של Chrome) |

**תצורת nginx proxy_cache (מושגית):**
```
proxy_cache_path /var/cache/nginx levels=1:2
    keys_zone=tactitok:10m
    max_size=10g
    inactive=30d
    use_temp_path=off;

# שמירת קבצי תוכן אגרסיבית (גדולים, משתנים לעתים נדירות)
location /api/content/ {
    proxy_pass https://CLOUD_SERVER;
    proxy_cache tactitok;
    proxy_cache_valid 200 30d;
    proxy_cache_use_stale error timeout updating http_502 http_503 http_504;
    add_header X-Cache-Status $upstream_cache_status;
}

# שמירת מטה-נתוני קטלוג עם תוקף קצר יותר
location /api/catalog {
    proxy_pass https://CLOUD_SERVER;
    proxy_cache tactitok;
    proxy_cache_valid 200 5m;
    proxy_cache_use_stale error timeout updating http_502 http_503 http_504;
    add_header X-Cache-Status $upstream_cache_status;
}

# SPA fallback routing
location / {
    root /var/www/spa;
    try_files $uri $uri/ /index.html;
}
```

### 8.2 פריסת מכשיר קצה

```
Edge Device (Windows PC with 10″ tablet display)
│

│       ├── Navigates to http://localhost:8080
│       ├── IndexedDB: device profile, download records, local actions

**תהליך בנייה ב-Docker:**
1. `client/` מכיל קבצי vanilla HTML/CSS/JS — ללא שלב בנייה נדרש

### 8.4 מודל רשת

```
Chrome (Windows) ──HTTP──► localhost:8080 (nginx in Docker/Linux VM)

                          Cloud Server:443 (nginx TLS → uvicorn:8000)

HQ Desktop (browser) ──HTTPS──► Cloud Server:443

## 9. סקירת זרימת נתונים / תוכן

### 9.1 זרימת העלאת תוכן

```
HQ Staff → Admin SPA → POST /api/admin/content (multipart) → API Server
    → אמת (סוג, גודל, פורמט)
    → ContentFileService.store(file) → ./data/content/{id}.{ext}
    → CatalogService.create(metadata) → PostgreSQL
    → החזר 201 + מטה-נתוני תוכן
```

*(ללא שינוי מ-v0.1 — העלאות עוברות ישירות לענן משולחן עבודה מטה)*

### 9.2 זרימת סנכרון מטה-נתונים (קצה)

```
Edge SPA (בפתיחת אפליקציה / רענון ידני)
    → GET /api/catalog → localhost:8080 (פרוקסי קצה)
    → פרוקסי קצה בודק מטמון:
        → MISS או STALE: העברה לשרת ענן → שמירה במטמון → החזרה
        → HIT: הגשה ממטמון
        → ענן לא נגיש: הגשת מטמון ישן (proxy_cache_use_stale)
    → Edge SPA מקבל catalog JSON
    → שומר ב-IndexedDB לסינון/חיפוש בצד לקוח
    → ממשק מוצג מנתוני קטלוג מנותחים
```

### 9.3 זרימת צריכת תוכן (וידאו)

```
Edge SPA (משתמש גולל לוידאו בריילס)
    → <video src="/api/content/{id}/file">
    → דפדפן שולח GET עם Range header ל-localhost:8080
    → פרוקסי קצה:
        → אם שמור במטמון: הגשה מ-/var/cache/nginx/ (HIT)
        → אם לא שמור: פרוקסי לענן, שמירת תגובה מלאה במטמון, סטרימינג ל-Chrome
    → דפדפן מנגן וידאו בצורה פרוגרסיבית
```

**הערה:** nginx `proxy_cache` שומר את התגובה המלאה בבקשה הראשונה. בקשות עוקבות (כולל range requests לאותו קובץ) מוגשות ממטמון. משמעות הדבר שנגינה ראשונה של וידאו גם שומרת אותו מראש לשימוש אופליין.

### 9.4 זרימת צריכת תוכן (PDF)

```
Edge SPA (משתמש פותח PDF מהספרייה)
    → fetch("/api/content/{id}/file") → localhost:8080
    → פרוקסי קצה מגיש ממטמון או מפרוקסי לענן (כמו וידאו)
    → Edge SPA מעביר ArrayBuffer ל-PDF.js
    → PDF.js מציג עמוד-אחר-עמוד ב-canvas
```

### 9.5 זרימת הורדה

```
Edge SPA (משתמש מקיש הורדה)
    → fetch("/api/content/{id}/file") → localhost:8080
    → פרוקסי קצה:
        → אם שמור: הגשה מיידית (HIT)
        → אם לא שמור: פרוקסי לענן → שמירה → סטרימינג ל-Chrome
    → SPA מקבל תגובה מלאה (קורא עד השלמה כדי להבטיח שמירה בפרוקסי)
    → SPA שומר רשומת הורדה ב-IndexedDB (id, title, type, size, date, version)
    → ממשק מתעדכן: פריט מופיע בלשונית הורדות
```

**מה השתנה (v0.2):** ה-SPA אינו עוד שומר קובץ ב-Cache API. הקובץ שמור על ידי פרוקסי הקצה. ה-SPA שומר רק רשומת מטה-נתונים ב-IndexedDB למעקב אחר מה שהמשתמש הוריד במפורש. מטמון הפרוקסי מחזיק את הקובץ בפועל.

### 9.6 זרימת נגינה אופליין

```
Edge SPA (משתמש פותח פריט, ענן לא נגיש)
    → קריאת רשומת הורדה מ-IndexedDB (מטה-נתונים בלבד)
    → בקשת קובץ: <video src="/api/content/{id}/file"> או fetch(...)
    → בקשה עוברת ל-localhost:8080 (פרוקסי קצה)
    → פרוקסי קצה: ענן לא נגיש → הגשה ממטמון (STALE)
    → דפדפן מנגן וידאו / SPA מציג PDF כרגיל
```

**מה השתנה (v0.2):** נגינה אופליין היא שקופה. ה-SPA מגיש אותה בקשה כמו אונליין; הפרוקסי מטפל בהגשה אופליין. ללא נתיב קוד אופליין מיוחד ב-SPA (ללא cache API lookups, ללא blob URLs). קוד ה-SPA לנגינה אונליין ואופליין זהה.

### 9.7 זרימת טעינה מוקדמת (ריילס)

```
Edge SPA (בזמן שהוידאו הנוכחי מנוגן)
    → זיהוי הוידאו הבא בפיד
    → fetch("/api/content/{nextId}/file", { headers: { Range: "bytes=0-{N}" } })
    → בקשה עוברת ל-localhost:8080 (פרוקסי קצה)
    → פרוקסי: אם לא שמור, שולף קובץ מלא מהענן ושומר
    → מחזיר תגובה חלקית ל-Chrome
    → שמירת תגובה חלקית בזיכרון
    → כשמשתמש מחליק לוידאו הבא:
        → אתחול נגינה מה-bytes שנטענו מוקדם
        → המשך דרך בקשות נוספות (מוגשות ממטמון פרוקסי — מיידי)
```

**יתרון הפרוקסי:** בקשת הטעינה המוקדמת מפעילה את הפרוקסי לשמור את הקובץ המלא. בקשות range עוקבות לאותו קובץ מוגשות ממטמון עם latency אפס.

## 10. גישת אופליין / סנכרון / מתמידות מקומית

### 10.1 הקצאת אחסון

| מאגר | מיקום | תכנים | ציפיית גודל | מתמידות |
|------|-------|--------|------------|--------|
| **מטמון פרוקסי קצה** | מערכת קבצים Docker (Linux VM) | קבצי תוכן שמורים + תגובות API | עד 10 GB | נמשך עד ניקוי Docker volume או פינוי מטמון |
| **IndexedDB** | Chrome (Windows) | פרופיל מכשיר (עניינים), רשומות הורדה (מטה-נתונים), פעולות מקומיות (לייק/שמירה) | <1 MB | נמשך בין הפעלות |
| **זיכרון** | Chrome (Windows) | חיצוני טעינה מוקדמת לוידאו הבא; נתוני קטלוג מנותחים | ~5–20 MB | אובד בסגירת דף |

**מה השתנה (v0.2):** Cache API ו-Service Worker cache אינם בשימוש עוד. `proxy_cache` של פרוקסי הקצה מחליף את שניהם. זה מבטל את כל קוד Service Worker וקוד Cache API מה-SPA, ומפשט משמעותית את הלקוח.

### 10.2 מטריצת יכולת אופליין

| תכונה | אונליין | אופליין (ענן לא נגיש) | כיצד |
|-------|---------|---------------------|------|
| אתחול אפליקציה | טעינת SPA מפרוקסי | ✅ SPA מקומי (ארוז ב-Docker) | nginx מגיש קבצים סטטיים מ-`/var/www/spa/` |
| פיד ריילס | סטרימינג דרך פרוקסי → ענן | ✅ אם נגיש בעבר | פרוקסי מגיש קבצי וידאו שמורים |
| עיון בספרייה | קטלוג טרי מענן דרך פרוקסי | ✅ קטלוג שמור | פרוקסי מגיש תגובת `/api/catalog` שמורה |
| נגינת וידאו | סטרימינג דרך פרוקסי → ענן | ✅ אם נגיש בעבר או הורד | פרוקסי מגיש קובץ שמור |
| צפיית PDF | שליפה דרך פרוקסי → ענן | ✅ אם נגיש בעבר או הורד | פרוקסי מגיש קובץ שמור |
| לשונית הורדות | הצגת רשומות הורדה | ✅ תמיד | רשומות IndexedDB; קבצים מוגשים ממטמון פרוקסי |
| שינוי עניינים | מיידי | ✅ מקומי בלבד | כתיבה ל-IndexedDB |
| סנכרון מטה-נתונים | פרוקסי מעביר לענן | ❌ הגשת מטמון ישן | הצגת חותמת "סונכרן לאחרונה"; פרוקסי מגיש ישן |
| חיפוש | סינון בצד לקוח על קטלוג | ✅ על קטלוג שמור | סנפשוט קטלוג IndexedDB |

**שיפור מרכזי (v0.2):** פיד ריילס אופליין אפשרי כעת לוידאו שנגיש בעבר. ב-v0.1, ריילס דרש שידור חי. כעת מטמון הפרוקסי הופך כל וידאו שנצפה בעבר לזמין אופליין.

### 10.3 אסטרטגיית סנכרון

- **כיוון:** פרוקסי קצה → pull ענן (שקוף; פרוקסי קצה מעביר בקשות)
- **טריגר:** בפתיחת אפליקציה + כפתור רענון ידני (SPA מבקש `/api/catalog`)
- **מטען:** קטלוג JSON מלא (כל מטה-נתוני תוכן)
- **פתרון התנגשויות:** השרת מנצח; Edge SPA מחליף קטלוג מקומי ב-IndexedDB
- **ישנות:** פרוקסי קצה שומר קטלוג ~5 דקות; אחרי כן, הבקשה הבאה עוברת לענן (אם נגיש)
- **סנכרון מצטבר:** לא ב-MVP (הקטלוג קטן: ~15 פריטים). עיצוב API לתמוך בפרמטר `?since=timestamp` לסנכרון דלתא עתידי.

### 10.4 ניהול מטמון פרוקסי קצה

| הגדרה | ערך | נימוק |
|-------|-----|-------|
| **גודל מטמון מקסימלי** | 10 GB | כל תוכן הדמו (~15 פריטים × ≤100 MB) מתאים בנוחות |
| **תוקף מטמון קבצי תוכן** | 30 יום | תוכן משתנה לעתים נדירות; בדיקת גרסה בסנכרון מטה-נתונים מזהה עדכונים |
| **תוקף מטמון קטלוג** | 5 דקות | תכוף מספיק לדמו; הגשה ישנה כשאופליין |
| **פינוי לא-פעיל** | 30 יום | פריטים שלא נגישים ל-30 יום מפונים; נדיב לדמו |
| **הגשה ישנה** | בשגיאה, timeout, 502, 503, 504 | מבטיח פונקציונליות אופליין |

## 11. שיקולי אבטחה / אמון / תפעול

### 11.1 אבטחת תעבורה

- ענן: כל התעבורה דרך HTTPS (TLS 1.2+); nginx מסיים TLS על פורט 443
- פרוקסי קצה → ענן: HTTPS (פרוקסי מאמת אישור שרת ענן)
- Chrome → פרוקסי קצה: HTTP דרך localhost (ללא TLS — אותו מחשב; ללא חשיפת רשת)
- אישור חתום-עצמי או Let's Encrypt לשרת ענן בדמו

### 11.2 אימות

| משטח | מנגנון | יישום |
|------|--------|-------|
| Edge SPA → פרוקסי קצה → Content API | ללא (ציבורי, קריאה-בלבד) | ללא auth middleware ב-`/api/catalog`, `/api/content` |
| Admin SPA → Admin API | סיסמה משותפת | `POST /api/admin/login` → מחזיר token פעלה (JWT או opaque); token נשלח ב-header `Authorization` |
| פעלת אדמין | אימות בצד שרת | תפוגת token; נשמר ב-sessionStorage (לא localStorage — מתנקה בסגירת כרטיסייה) |

### 11.3 אבטחת תוכן

- קבצי תוכן מוגשים על ידי שרת ה-API — ללא כתובות URL ישירות למערכת קבצים חשופות
- מטמון פרוקסי קצה נמצא במערכת קבצים Linux VM המקומית — לא נגיש ישירות מ-Windows (Chrome ניגש דרך HTTP בלבד)
- העלאת קבצים מאמת MIME type + סיומת קובץ
- לא מתקבלים סוגי קבצים הניתנים להרצה
- קבצים שהועלו נשמרים עם שמות content-id שנוצרו (ללא נתיבים שסופקו על ידי משתמש)

### 11.4 יסודות תפעוליים

- לוגינג שרת רגיל (לוגי בקשות, לוגי שגיאות) — ללא פלטפורמת אנליטיקה
- connection pooling ל-PostgreSQL (גודל pool ברירת מחדל מספיק ל-~20 במקביל)
- systemd לrestart אוטומטי של תהליך uvicorn
- מדיניות Docker restart `always` ל-container פרוקסי קצה
- ללא פלטפורמת ניטור/התראה ב-MVP

| 5 | דילוג על Docker; התקנה ידנית של nginx + קבצי SPA על Linux VM | מסיר עקומת למידה Docker; קשה יותר לעדכן/לשחזר |

## 19. הערות להמשך

הנחיות לצוות ממשיך המרחיב מעבר לארכיטקטורת MVP:

- **מעבר ל-microservices:** שכבת השירותים (CatalogService, ContentFileService, וכו') היא מודולרית בפייתון. כל שירות ניתן לחילוץ לתהליך FastAPI נפרד אם גדילה דורשת זאת. התחל בחילוץ ContentFileService אם עומס וידאו גדל.
- **אינטגרציית CDN:** פרוקסי התוכן (`/api/content/{id}/file`) ניתן לסינון ב-CDN. הוסף headers מטמון (`Cache-Control`, `ETag`) מהיום הראשון כך שאינטגרציית CDN היא שינוי תצורה, לא שינוי קוד.
- **סנכרון דחיפה (WebSocket/SSE):** FastAPI תומך ב-WebSocket ו-SSE באופן נייטיב. פרוקסי הקצה יעביר חיבורי WebSocket ללא שמירה במטמון. מודול הסנכרון של ה-SPA צריך להיות מעוצב עם ממשק `SyncProvider` כך שpull ניתן להחלפה ב-push.
- **אחסון אובייקטים (S3/MinIO):** להפשטת האחסון (`ContentFileService`) יש מתודות `store`, `retrieve`, `delete`. החלף את יישום מערכת הקבצים ב-boto3/S3 client תוך שמירת אותו ממשק.
- **אימות משתמשים:** הוסף FastAPI dependency injection לאימות. ה-API כבר מפריד בין נתיבים ציבוריים (קצה) למוגנים (ניהול). עתידי: הוסף auth לנתיבי קצה, החלף device-id ב-user-id.
- **Clustering / load balancing:** gunicorn עם workers מרובים או מספר instances uvicorn מאחורי nginx upstream. PostgreSQL מטפל בחיבורים במקביל באופן נייטיב. מאגר הקבצים משותף דרך מערכת קבצים (או מאגר אובייקטים למולטי-VM).
- **CI/CD:** מאגר שטוח תומך בצינור פשוט: lint → test → build Docker image → deploy. הוסף GitHub Actions או דומה כשהצוות מוכן.
- **אסטרטגיית database migration:** שימוש ב-Alembic migrations. כל שינוי schema הוא קובץ migration בגרסה. זה קריטי-להמשך — חייב להיות מוגדר מספרינט 1.
- **שיפור פרוקסי קצה:** הוסף prefetch תוכן בצד-פרוקסי (background job שמושך כל תוכן חדש מהענן למטמון). כרגע הפרוקסי שומר בגישה ראשונה; prefetch יזום מבטיח שכל התוכן זמין אופליין לפני שהמשתמש מבקש אותו.
- **Service Worker (עתידי):** אם דרישות קצה גדלות מעבר למה שהפרוקסי מטפל (למשל, push notifications, background sync של פעולות מקומיות), Service Worker ניתן להוספה ל-SPA לצד הפרוקסי. הפרוקסי מספק את בסיס השמירה במטמון; SW יוסיף תכונות ברמת דפדפן.
