## 1. מטרת מסמך זה

מסמך זה מגדיר את גבול מערכת TactiTok למטרות MVP:

- מה בתוך המערכת לעומת מחוצה לה
- אילו גורמים מתקשרים איתה
- אילו שירותים חיצוניים או תשתיות מונחות כקיימות
- מה רץ על מכשיר הקצה, מה רץ על השרת, ומה מחוץ לגבול המוצר
- היכן נמצאים גבולות האמון, הרשת והפעולה

המסמך נועד לצמצם עמימות לפני כתיבת מפרט MVP, ארכיטקטורת מערכת, מודל נתונים וחוזה API. כל מסמך נגזר חייב לכבד גבולות אלו.

## 2. המערכת המוגדרת

**TactiTok MVP** הוא יישום web דו-משטחי עם ארכיטקטורת קצה תלת-שכבתית:

- **מכשיר קצה** — מחשב Windows עם תצוגת טאבלט 10". Linux VM מקומי מריץ Docker עם פרוקסי מטמון nginx + חבילת Edge SPA. Chrome על Windows רץ במצב קיוסק ומתחבר ל-`localhost`. הפרוקסי שומר תוכן מהענן לגישה אופליין.
- **שרת ענן** — VM ענן המארח את Content API, פורטל הניהול, מסד הנתונים של מטה-נתונים ואחסון קבצי תוכן.
- **שולחן עבודה מטה** — דפדפן רגיל הגולש לפורטל הניהול בשרת הענן.

גבול המערכת כולל הכל שהצוות בונה ופורס: Edge SPA, פרוקסי הקצה (Dockerfile + תצורת nginx), שרת API בענן, Admin SPA, ו-schema של מסד הנתונים. הוא אינו כולל OS הבסיסי, תצורת קיוסק, הקמת Linux VM, הקמת תשתית ענן, ואף מערכת ארגונית חיצונית.

## 3. הגדרות / מונחים

| מונח | הגדרה |
|------|-------|
| **לקוח קצה** | ה-SPA הרץ ב-Chrome על Windows; מתחבר ל-localhost (פרוקסי קצה), לעולם לא ישירות לענן |
| **פרוקסי קצה** | פרוקסי הפוך עם מטמון nginx הרץ ב-Docker ב-Linux VM של מכשיר הקצה; משרת את ה-SPA ומפרוקסי/שומר תוכן ענן במטמון |
| **שרת** | VM הענן המארח את ה-API, פורטל הניהול, אחסון תוכן |
| **פורטל ניהול** | ממשק web המוגש על ידי השרת לצוות מטה לניהול תוכן וקטגוריות |
| **Content API** | ממשק HTTP/REST שלקוח הקצה משתמש בו לשליפת מטה-נתונים וקבצי תוכן |
| **מאגר תוכן** | מיקום בצד השרת שבו קבצי וידאו ו-PDF שהועלו נשמרים |
| **קטלוג מטה-נתונים** | הנתונים המובנים (כותרות, קטגוריות, תגיות, גרסאות) המתארים את כל פריטי התוכן |
| **פרופיל מכשיר** | תצורה מקומית-לדפדפן במכשיר הקצה (בחירת עניינים); ללא זהות בצד השרת |
| **מטמון מקומי** | מטמון פרוקסי קצה (nginx `proxy_cache` לתוכן/מטה-נתונים) + IndexedDB ב-Chrome (לפרופיל מכשיר, רשומות הורדה, פעולות מקומיות) |
| **גבול אמון** | הקו הלוגי המפריד בין רכיבים פנימיים מהימנים לחיצוניים לא-מהימנים |

## 5. מערכות/רכיבים בתחום

אלו נבנים ובבעלות צוות TactiTok:

### 5.1 לקוח קצה (Browser SPA)

| רכיב | אחריות |
|------|--------|
| **פיד ריילס** | פיד וידאו אנכי-גלילה עם סינון עניינים |
| **תצוגת ספרייה** | עיון בעץ קטגוריות + חיפוש טקסט |
| **נגן וידאו** | נגינה בדפדפן עם התנהגות טעינה-מוקדמת-ראשונה |
| **מציג PDF** | עיצוב PDF בדפדפן עמוד-אחר-עמוד (למשל, PDF.js) |
| **מנהל הורדות** | הורדה ביוזמת משתמש מפעילה שמירת קובץ תוכן במטמון פרוקסי קצה; רשומת הורדה (מטה-נתונים בלבד) נשמרת ב-IndexedDB; רשימה + מחיקה |
| **פרופיל מכשיר** | בחירת עניינים מקומית, שמורה ב-IndexedDB |
| **לייק/שמירה מקומיים** | נשמר מקומית במכשיר; ללא סנכרון שרת ב-MVP |
| **לקוח סנכרון מטה-נתונים** | מושך קטלוג עדכני מהשרת בפתיחת אפליקציה / רענון ידני |
| **נגינה אופליין** | משרת תוכן שהורד בעבר ממטמון מקומי כשהרשת אינה זמינה |

### 5.2 שרת

| רכיב | אחריות |
|------|--------|
| **Content API** | נקודות קצה REST ל: מטה-נתוני קטלוג, הגשת קבצי תוכן, חיפוש, עץ קטגוריות |
| **Admin API** | נקודות קצה REST ל: העלאת תוכן, CRUD מטה-נתונים, CRUD קטגוריות, ניהול עניינים/תגיות |
| **פורטל ניהול** | ממשק web לצוות מטה (מוגש על ידי אותו שרת) |
| **מאגר תוכן** | אחסון קבצים בצד שרת לוידאו ו-PDF שהועלו (החלטה על מאגר אובייקטים חיצוני נדחית) |
| **מאגר מטה-נתונים** | מסד נתונים המחזיק קטלוג תוכן, קטגוריות, עניינים ומידע גרסאות |
| **שרת נכסים סטטיים** | משרת חבילת Admin SPA (חבילת Edge SPA נמצאת ב-Docker במכשיר קצה) |

### 5.3 פרוקסי קצה (Docker על Linux VM)

| רכיב | אחריות |
|------|--------|
| **פרוקסי קצה** (nginx ב-Docker) | משרת Edge SPA מקבצים סטטיים מחוברים לתמונה; מפרוקסי + שומר בקשות API/תוכן לשרת ענן במטמון; משרת תוכן שמור במטמון אופליין |

### 5.4 משותף / חוצה-חתך

| רכיב | אחריות |
|------|--------|
| **סיום TLS** (ענן) | HTTPS לתקשורת אדמין ↔ שרת ופרוקסי קצה ↔ שרת |
| **משלוח תוכן** | תמיכת HTTP range request לסטרימינג / טעינה מוקדמת של וידאו |

## 6. מערכות/רכיבים מחוץ לתחום

אלו **אינם נבנים** ב-MVP. הארכיטקטורה לא תחסום הוספתם בעתיד, אך אין קוד, API, או schema שנוצרים עבורם כעת.

| רכיב | מדוע מחוץ לתחום |
|------|----------------|
| **זהות משתמש / אימות** | ה-MVP הוא קיוסק/אנונימי; פרופיל מכשיר בלבד |
| **מנוע RBAC / הרשאות** | ללא זהות משתמש → ללא תפקידים |
| **מנוע המלצות** | ה-MVP משתמש בסינון לפי עניינים בלבד |
| **שירות אנליטיקה / טלמטריה** | ללא דיווח שימוש ב-MVP |
| **שירות התראה / סנכרון דחיפה** | ה-MVP משתמש בסנכרון מטה-נתונים מבוסס-pull |
| **צינור עיבוד וידאו** | ה-MVP מניח קבצי MP4 מוכנים להעלאה |
| **CDN / שכבת מטמון קצה** | שרת יחיד משרת תוכן ישירות; ~20 מכשירים |
| **SSO חיצוני / LDAP / IAM ארגוני** | ללא אינטגרציה עם מערכות זהות חיצוניות |
| **DRM / הגנת תוכן** | אבטחה דרך בידוד רשת + TLS בלבד |
| **צ'אט / חברתי / תגובות** | ללא תכונות חברתיות |
| **ריצת i18n / רב-שפות** | שפה אחת; ארכיטקטורת מפתח-מחרוזת בלבד |
| **אכיפת מגבלת אחסון** | ללא מגבלה ב-MVP; מודל נתונים עוקב אחר גדלים לשימוש מאוחר |
| **ניהול גרסאות מורכב** | החלפה במקום עם תג "עודכן"; ללא rollback |
| **זרימות אישור לתוכן** | העלאה → פרסום מיידי; ללא שלב סקירה |
| **פלטפורמת ניטור / התראה / observability** | לא נבנה; רק לוגי שרת רגילים |

## 7. מערכות / תלויות חיצוניות

אלו **אינם נבנים על ידי הצוות** אך מונחים כקיימים ופועלים:

| תלות | מה היא מספקת | אחריות הצוות |
|------|-------------|-------------|
| **ספק VM ענן** | Linux VM עם IP ציבורי/פרטי, CPU/RAM/דיסק מספיק ל-~20 מכשירי קצה במקביל | הקמה ותצורה; לא בניה |
| **DNS / דומיין** (אופציונלי) | שם מארח לשרת; ניתן להשתמש ב-IP גולמי לדמו | תצורה אם זמין |
| **אישור TLS** | HTTPS לכל התעבורה | קבלה והתקנה (Let's Encrypt או חתום-עצמי לדמו) |
| **דפדפן Chrome** | סביבת ריצה ללקוח קצה; מצב קיוסק מתוצר כבר | בניה עבור Chrome; לא תצורת קיוסק |
| **Docker** | סביבת ריצה קונטיינרים ב-Linux VM של מכשיר הקצה | הצוות כותב Dockerfile + nginx.conf; מניח Docker daemon מותקן |
| **Windows + Linux VM על טאבלט** | סביבת OS במכשיר הקצה; Linux VM מארח Docker | לא מנוהל על ידי הצוות; הנח שעובד; Docker daemon מונח כמותקן |
| **תשתית רשת** | קישוריות בין מכשירי קצה ל-VM ענן | לא מנוהל; הנח שקיים (עלול להיות לא יציב) |
| **קורפוס תוכן לדמו** | 10 PDF + 5 וידאו (≤3 דקות, MP4, מוכן להעלאה) | הצוות או בעלי עניין מכינים תוכן; המערכת אינה מייצרת אותו |

## 9. גבולות רשת ואופליין

### 9.1 מודל קישוריות

| מצב | התנהגות לקוח קצה | התנהגות שרת |
|-----|----------------|------------|
| **אונליין** | משיכת קטלוג מטה-נתונים; סטרימינג/שליפת תוכן לפי דרישה; הורדת פריטים | משרת בקשות API כרגיל |
| **מדורדר** | טעינה מוקדמת של תחילת וידאו (N שניות ראשונות); שחיקה חלקה עם מחווני טעינה; מטה-נתונים עלולים להיות ישנים | משרת עם תגובות איטיות יותר; הלקוח מטפל בפסקי זמן |
| **אופליין** | משרת רק תוכן שהורד מקומית + מטה-נתונים שמורים במטמון; אין תוכן חדש זמין; ללא סנכרון מטה-נתונים | לא נגיש |

### 9.2 זרימת נתונים לפי קישוריות

| סוג נתון | אונליין | אופליין | מיקום אחסון |
|----------|---------|---------|------------|
| **קטלוג מטה-נתונים** | נמשך מהשרת | עותק מטמון מקומי בשימוש | DB שרת → Edge IndexedDB |
| **תוכן וידאו** | מסוטרם מהשרת דרך פרוקסי קצה (טעינה מוקדמת + range requests) | מוגש ממטמון פרוקסי קצה (אם נגיש בעבר או הורד) | מאגר קבצים שרת → מטמון פרוקסי קצה → Chrome |
| **תוכן PDF** | נשלף מהשרת דרך פרוקסי קצה | מוגש ממטמון פרוקסי קצה (אם נגיש בעבר או הורד) | מאגר קבצים שרת → מטמון פרוקסי קצה → Chrome |
| **פרופיל מכשיר** | מקומי בלבד (לעולם לא נשלח לשרת) | מקומי בלבד | Edge IndexedDB |
| **פעולות לייק/שמירה** | מקומי בלבד ב-MVP | מקומי בלבד | Edge IndexedDB |

### 9.3 מנגנון סנכרון (MVP)

- **כיוון:** משיכת קצה → שרת בלבד (ללא דחיפה)
- **טריגר:** בפתיחת אפליקציה + כפתור רענון ידני
- **היקף:** קטלוג מטה-נתונים מלא (מטען קטן עבור ~15 פריטים; ניתן לדפדוף מאוחר יותר)
- **אסטרטגיית התנגשות:** השרת סמכותי; הקצה מחליף מטמון מטה-נתונים מקומי בסנכרון

## 10. גבולות אמון / אבטחה

### 10.1 אזורי אמון

```
┌──────────────────────────────────────────┐
│  אזור 1 — שרת (מהימן)                    │
│  - Content API                            │
│  - Admin API (מוגן בסיסמה)               │
│  - DB מטה-נתונים                          │
│  - מאגר קבצי תוכן                         │
│  - כל הנתונים סמכותיים כאן               │
└────────────────┬─────────────────────────┘
                 │ TLS (HTTPS)
                 │ תעבורה לא-מהימנה
┌────────────────┴─────────────────────────┐
│  אזור 2 — מכשיר קצה (חצי-מהימן)          │
│  - ללא זהות; פרופיל מכשיר בלבד           │
│  - פרוקסי קצה שומר נתוני ענן מקומית      │
│  - Chrome מתחבר לפרוקסי דרך localhost    │
│  - יכול לקרוא קטלוג + תוכן              │
│  - לא יכול לשנות נתוני שרת               │
│  - אחסון מקומי בשליטת משתמש              │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  אזור 3 — לקוח ניהול (מאומת)            │
│  - פעלת אימות סיסמה                      │
│  - יכול להעלות תוכן                      │
│  - יכול לשנות קטלוג (CRUD)               │
│  - רץ ברשת מטה מהימנה                    │
└──────────────────────────────────────────┘
```

### 10.2 החלטות אבטחה ל-MVP

| גבול | גישת MVP |
|------|---------|
| **תעבורה** | TLS (HTTPS) לכל התעבורה — קצה ↔ שרת ואדמין ↔ שרת |
| **אימות לקוח קצה** | ללא — אנונימי/קיוסק; כל תוכן הקטלוג קריא |
| **אימות פורטל ניהול** | סיסמה משותפת פשוטה (פעלת שרת או token) |
| **רגישות תוכן** | הנח רגיש; צמצם חשיפה; אחסן מקומית רק מה שהמשתמש הוריד במפורש |
| **גישת שרת** | הגבל לרשת ידועה אם אפשרי; ללא API פתוח ציבורי |
| **אחסון מקומי** | מנוהל-דפדפן; ללא הצפנה ב-MVP (ארגז חול הדפדפן הוא הגבול) |
| **טלמטריה** | ללא ב-MVP; אין נתוני שימוש שיוצאים ממכשיר הקצה |

### 10.3 מה הצוות אינו מאבטח

- אבטחת OS ברמת הטאבלט (תצורת קיוסק חיצונית)
- חוקי חומת אש של VM ענן (צוות תשתיות / הקמה)
- אבטחה פיזית של מכשיר
- סיווג תוכן או DRM

## 15. הערות להמשך

הנחיות לצוות ממשיך המרחיב מעבר לגבולות MVP:

- **זהות משתמש:** דפוס פרופיל-מכשיר-אנונימי צריך להיות ניתן להחלפה ב-user-id. ה-Content API צריך לקבל header אימות אופציונלי שה-MVP מתעלם ממנו אך post-MVP אוכף.
- **סנכרון דחיפה:** סנכרון מטה-נתונים מבוסס-pull ניתן להוספת ערוץ WebSocket/SSE. שכבת ה-API צריכה להיות מעוצבת כך שהדחיפה תהיה תוספת, לא שכתוב.
- **העברת מאגר תוכן:** אם הצוות מתחיל עם מערכת קבצים מקומית, שכבת האחסון צריכה להיות מאחורי ממשק (קרא/כתוב/מחק לפי content-id) כך שניתן להחליף S3/MinIO.
- **CDN / מטמון קצה:** בקנה-מידה גבוה יותר (>20 מכשירים), CDN או מטמון פרוקסי הפוך לפני מאגר התוכן מפחית עומס שרת. ה-API צריך להגיש תוכן עם headers מטמון נכונים מהיום הראשון.
- **עיבוד וידאו:** אם נדרשות העלאות שאינן MP4, ניתן להוסיף תור משימות רקע (עובד ffmpeg) מאחורי נקודת קצה ההעלאה. ה-API להעלאה צריך לאמת פורמט ולדחות קבצים לא נתמכים ב-MVP.
- **אנליטיקה:** פעולות לייק/שמירה נשמרות מקומית ב-MVP. משימת סנכרון עתידית יכולה להעלות אותן בסבב לשרת כשזהות קיימת. Schema האחסון המקומי צריך לכלול חותמות-זמן ו-content-ids כדי לאפשר זאת.
- **ריבוי-שוכרים:** ה-MVP הוא חד-שוכר. אם ארגונים מרובים זקוקים לקטלוגים נפרדים, מודל הנתונים צריך להשתמש בעמודת tenant-id שה-MVP מגדיר לקבוע.
